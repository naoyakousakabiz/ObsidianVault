## 0.2 用語ミニ辞典（このファイル内の言葉）

| 用語 | ひとこと |
|:--|:--|
| **North Star** | 「成功した」と言える状態を**1文**で表したもの。数字が無くてもよいが、後でKPIと接続する。 |
| **ファクトベース** | 感想ではなく、**観測・記録・証跡**に立つこと。「まだ測ってない」もファクトとして書く。 |
| **アウトプットカード** | 判断・観測・設計を**1枚に固定**したメモ。議事録の代わりに、後から再利用する。 |
| **PoC（概念実証）** | 小さく作って「効くか」を確かめる工程。失敗が許容される**実験**であることをクライアントと合意する。 |
| **ガードレール** | AIにしてよいこと／してはいけないことの**境界**（人の承認、参照データ、露出する情報）。 |
| **データ正本** | 数字・顧客情報・在庫など、「これが正」と決める**単一の源**。複数あると自動化が壊れる。 |
| **冪等（べきとう）** | 同じ処理を繰り返しても**結果が壊れない**こと（通知二重など）。 |

## 0.3 7ブロックを一周するミニ例（イメージ用）

**設定:** 中小B2Bの「受注後の初回連絡」が遅い・漏れるというクレーム。

1. **ブロック1** — North Star：「受注から24時間以内に初回連絡が**100%**」／非目的：「CRMの抜本置換はしない」  
2. **ブロック2** — ファクト：実測で平均**36時間**、欠損**月5件**（未計測なら「未計測」と書き、取得日を決める）  
3. **ブロック3** — ボトルネック：承認待ち、担当者不明、テンプレが分散  
4. **ブロック4** — 原則：テンプレ統一・承認は1段階・顧客メールは人が最終送信  
5. **ブロック5** — AIは「下書き生成のみ」、正本はCRM  
6. **ブロック6** — Wave1でテンプレ＋チェックリスト、Wave2で下書きツール試験導入  
7. **ブロック7** — KPI：24時間以内率、欠損件数、品質ゲート通過率（定義はチームで合意）  

この流れ自体が**顧客への説明ストーリー**になる。

## 2. スコープと前提（エンゲージメントの枠）

納品物を書く前に、**やること／やらないこと**を固定する。

**合意すべき項目（チェックリスト）**

- [ ] **対象業務** — どのプロセス・部門か（境界の言い方まで）  
- [ ] **対象外** — 今回は触れない領域（システム刷新、組織再編など）  
- [ ] **ステークホルダー** — 最終決定者、現場責任者、情シス／法務の関与  
- [ ] **期間と節目** — 例：現状把握2週 → 設計4週 → パイロット4週  
- [ ] **成功の定義** — 定量（時間・コスト・ミス・速度）と定性（顧客体験・リスク）  

**ステークホルダー（軽量RACIの例）**

| 役割 | 名前または役職 | R / A / C / I |
|:--|:--|:--|
| スポンサー |  | A（説明責任） |
| 現場オーナー |  | R（遂行） |
| 情シス |  | C / I |
| コンサル |  | R / C |

R＝実行者、A＝説明責任、C＝相談先、I＝情報共有。**1つのA**に絞れるとブレが減る。

**アウトプットカード（複製先で埋める）**

```markdown
## アウトプットカード: スコープ合意

- 案件名:
- 期間:
- 対象業務（1〜3行）:
- 対象外（箇条書き）:
- 成功の定義（定量・定性）:
- 最終決定者（A）:
- RACIメモ:
```

## 5. ブロック1 — 成果定義と制約

**問い**

- 経営・現場が**本当に欲しい成果**は何か（コスト・速度・品質・リスクのどれが主か）。  
- **下げてはいけない品質**（顧客SLA、法令、ブランド、監査）は何か。  
- **人が必ず残す判断**は何か（最終承認、例外、クレームなど）。  

**AI導入特有**

- 誤りのコストが高い領域は、AIを**下書き専用**に限定する等、**ガードレールを制約として書く**。  

**アウトプットカード**

```markdown
## アウトプットカード: 成果と制約

- North Star（1文）:
- 非目的（やらないこと3つまで）:
- 品質・コンプライアンス境界:
- 人の判断が残る箇所:
- AIの許容役割（例：下書きのみ／社内限定）:
- 顧客向け説明で使う比喩・ストーリー（任意・1文）:
```
