# RINGBELL CS改善設計（AI自動化以外も含む）

## 0. 位置づけ

- 本書は、`Ringwell_CS_AI自動化設計.md` と同列の「CS改善全体」設計ファイル。
- AI自動化の実装詳細は `Ringwell_CS_AI自動化設計.md` を正本とし、本書はCS運用・CS Ops・テックタッチ施策の設計を扱う。
- 施策実行時は `40_施策実装/` に1施策1ファイルで起票する。

## 1. 目的

- 成婚率最大化、休会/離脱率低下、CS品質の再現性向上を同時に実現する。
- 「担当者の頑張り依存」から「運用構造で成果を再現する状態」へ移行する。

## 2. 改善の主軸

### 2.1 CS Ops（運用設計）

- 業務をイベント起点で標準化（入会、公開、交際進展、ブレイク、復帰）
- 週次定例で改善点Top3を抽出し、テンプレ/手順へ反映
- KPIを結果指標と行動指標に分離して運用

### 2.2 テックタッチ化（非同期支援）

- 面談依存を下げるため、節目でのLINE配信を標準化
- 正本チャネルはLINE
- 対象イベント（初期）:
  1. 入会後1週間
  2. 入会後1ヶ月
  3. 入会後2ヶ月
  4. 休会復帰時
  5. 真剣交際ブレイク時（一次接触SLA: 72時間）

### 2.3 動画化（知識提供の資産化）

- 1本1テーマで、視聴後の行動（CTA）を必ず提示
- 制作工数は1本最大90分
- 初期5本を先に出し、反応データに基づき改善する

## 3. KPI設計（最小）

### 3.1 結果指標

- 成婚率
- 真剣交際率
- 休会率

### 3.2 行動指標

- 配信到達率（LINE）
- 視聴率/完了率（動画）
- 配信後7日以内の行動再開率
- 面談後Next Action実行率
- ブレイク後72時間以内一次接触率

## 4. 90日実行ロードマップ

### 0-30日

- 面談要約テンプレ統一
- 面談後フォロー文標準化
- ブレイク/復帰シナリオ文面確定

### 31-60日

- テックタッチ動画5本の台本作成
- LINE配信開始
- 改善点Top3の週次運用開始

### 61-90日

- 配信KPIレビュー
- 面談要約〜内部共有の半自動化連携
- 継続/停止/改善の判断と次四半期計画化

## 8. CS改善設計の統合視点（AI自動化 + CS Ops）

### 8.1 業務の「手動」「AI補助」「自動」分類

| 業務フロー | 現状 | AI導入後 | リスク・制約 |
|:--|:--|:--|:--|
| **初回挨拶文生成** | 手動（テンプレ + 編集） | AI草案 → 人間承認 → 送信 | 個人差・誤送信リスク → `RB-CS-004` で防止 |
| **日程調整（候補日あり/なし分岐）** | 手動（テンプレ選択 + 文面編集） | AI自動分岐 + 候補生成 → 人間ゲート | 見落とし → タスク化・Slack通知で補助 |
| **事前準備フォローと再催促** | 手動（3日・7日で複数通） | AI日数管理 + 文面生成 + 送信判定 | メッセージ疲れ → テンプレで柔軟化 |
| **面談ログ整形・内部共有** | 手動（文字起こし要約・項目抽出） | AI要約 + 次アクション抽出 → 人間確認 | 解釈のずれ → 確認フロー・学び蓄積で改善 |
| **決済確認・例外フォロー** | 手動（Slack/Gmail確認） | 確認チェックリスト可視化 → 人間判定 | データ不足 → スプレッドシート入力責任明示 |
| **テックタッチ配信（LINE動画）** | 非実施 → 計画中 | 節目イベント起点で自動判定 → 人間承認 → 配信 | メッセージ過多 → セグメント・頻度設計で抑制 |

### 8.2 メトリクス設計（結果指標 × 行動指標）

**結果指標（望む変化）**
- 成婚率：月ベース（ドメイン的な最上位KPI）
- 真剣交際率：月ベース（中間成果）
- 休会率：月ベース（リスク指標）

**行動指標（プロセス改善の監視）**
- 初回返信SLA達成率：日ベース
- 面談到達率：面談予定 → 実施の比率（日〜週ベース）
- テックタッチ到達率・視聴完了率：配信ごと（週ベース）
- 面談後Next Action実行率：面談 → 実行確認の比率（週ベース）
- ブレイク対応72時間以内率：事象 → 対応までの時間（日ベース）

### 8.3 改善サイクル（週次定例での回し方）

**毎週（月曜18:00定例）**
1. 行動指標の確認（ダッシュボード）
2. 改善点Top3の抽出（施策・テンプレ改善・学び）
3. 追加学びの記録（面談ログからのFAQ・要注意ケース）
4. テンプレ・文面の更新反映

**月次（4週ごと）**
1. 結果指標のレビュー
2. 改善効果（Before/After）の確認
3. 次月の改善テーマ決定
4. テックタッチ配信内容・セグメントの見直し

## 9. リスク・制約・撤退基準

### 9.1 AI導入時のリスク

| リスク | 発火条件 | 対策 |
|:--|:--|:--|
| **個人情報漏洩** | 会員氏名・連絡先をAIに入力 | 必ず匿名化・段階化（ID/状況のみ） |
| **文体の不適合** | 生成文面のトーンが敬語/冷たく感じる | 返信前ゲート（RB-CS-004）で人間確認必須 |
| **誤送信・誤判定** | AI自動送信・自動分類が間違う | HITL（承認点）を明示、フォールバック手動フロー用意 |
| **配信疲れ・離脱増加** | テックタッチ配信が多すぎる | 頻度上限・セグメント設計で抑制。配信後行動率を監視 |
| **レジストリ不整備** | 施策IDが漏れる・重複する | `00_施策一覧.md` を単一正本に |

### 9.2 撤退基準

- **RB-CS-001〜003（AI自動化基盤）:** 3週PoC で「テンプレ改善 + 工数削減」が出なければ継続検討
- **RB-CS-002（テックタッチ配信）:** 初期5本配信後、視聴完了率 < 20% なら内容改善 or 配信停止検討
- **RB-CS-005/006（ダッシュボード・資産化）:** 入力負荷 > 削減効果なら簡略化 or 停止

## 8. 追加AI施策（04_AI参照を踏まえた新規）

### 8.1 施策候補

#### D. 品質ゲート自動チェック（会員送信前）

- 内容: 送信文の「日付/締切/SLA/URL/敬語トーン」を自動検査し、NG時は送信停止
- 効果: 誤送信・期限誤記を抑制
- 工数: 中
- リスク: ルール過剰で送信遅延の恐れ
- 依存: テンプレ定義の固定、最終承認者の明示

#### E. 面談ログ -> ナレッジ資産化パイプライン

- 内容: 面談要約から「再利用テンプレ」「FAQ」「注意ケース」を抽出し蓄積
- 効果: コンシェルジュ教育と回答品質の平準化
- 工数: 中
- リスク: 匿名化漏れ
- 依存: 匿名化ルール、保存先フォーマット

#### F. 週次CSダッシュボード半自動更新

- 内容: 施策KPI（到達率/行動再開率/ブレイク72h対応率）を自動集計して週次定例に出力
- 効果: 勘ではなく数字で優先順位を決められる
- 工数: 中〜高
- リスク: 元データ不整合時に誤判断
- 依存: データ定義、手入力項目の統一

### 8.2 優先順位（PoC -> 本番）

1. D: 品質ゲート自動チェック（事故防止が先）
2. F: 週次CSダッシュボード半自動更新（意思決定速度改善）
3. E: 面談ログ資産化（中長期の再現性強化）
