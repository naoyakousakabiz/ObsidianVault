# Ringwell CS AI自動化設計（NotebookLM × Claude Code × 周辺AI）

## 0. 目的

`Ringwell_CS_業務分解_初版.md` を実行可能な自動化設計へ変換する。  
目標は、CS品質を落とさずに「毎日17:00-18:00の1時間運用」で回る業務構造を作ること。

## 1. 設計原則

1. 人間は判断、AIは処理を担当する
2. 1ツール1責任（役割を混ぜない）
3. 返信前は必ず品質ゲートを通す
4. 例外時は手動へフォールバックする
5. ログを残し、毎週改善する

## 2. ツール責務マップ

### NotebookLM（理解・検証レイヤー）

- 公式テンプレ、運用ルール、FAQ、過去対応ログをソース化
- ケース別に「どう返すべきか」を根拠付きで整理
- 定例前の難案件の論点整理

### Claude Code（実行・自動化レイヤー）

- Slack通知/フォーム入力をトリガーに文面草案を生成
- 分岐ルールに沿って日程調整・リマインド文を自動生成
- 公開前チェックリストの自動判定補助
- Slack内部共有文（面談メモ/Next Action）の整形

### Cursor（編集・統合レイヤー）

- 文面最終化、運用ルール更新、テンプレ管理
- 複数ケース横断の改善反映

### 補助ツール

- ChatGPT：PR文生成メーカー、返信草案補助
- Gemini：面談ログ要約
- Google Workspace（Calendar/Gmail/Sheet）：実行トリガーと業務データの正本
- Slack：運用通知・エスカレーション・内部共有

## 3. 対象業務と自動化優先順位

## Phase 1（即効性重視）

1. 日程調整文面生成（候補日あり/なし）
2. 3日前フォーム依頼 + 未提出再催促文生成
3. 面談後フォロー文生成（公開目標日/締切/初週目安）

## Phase 2（公開処理の安定化）

4. 決済確認チェック補助（Slack/Gmail確認手順のガイド化）
5. 公開前Yes/Noチェックリスト生成
6. 公開完了連絡（会員向けLINE + 内部Slack投稿文）生成

## Phase 3（運用品質の底上げ）

7. 相談チャットの一次分類（定型回答/要判断/要エスカレーション）
8. 週次定例の事前共有メモ自動下書き
9. 面談メモからNext Action抽出・整形

## 4. NotebookLM設計（Ringwell専用）

## 4.1 ノートブック構成

- ノートブック名：`Ringwell_CS_運用中枢`
- 章構成（推奨）
  - テンプレ群（初回挨拶、日程調整、リマインド、公開後連絡）
  - ルール群（SLA、分岐、エスカレーション）
  - 事例群（良い返信/悪い返信、難案件）
  - FAQ群（婚活ルール、活動相談）

## 4.2 ソース投入優先

1. 最新テンプレ全文
2. `Ringwell_CS_業務分解_初版.md`
3. 面談後連絡テンプレ
4. 公開処理マニュアル抜粋
5. 過去の相談チャット（匿名化）

## 4.3 NotebookLMの使い方（定例）

- 難案件受領時：
  - 「このケースは定型対応か要判断か」を判定
  - 「候補返信を3案（丁寧/標準/短文）」で出す
- 週次定例前：
  - 進展者・成婚者・難案件の要点を1ページ要約

## 5. Claude Code自動化設計

## 5.1 自動化単位（ジョブ）

### Job-A: 日程調整メッセージ生成

- 入力：候補日有無、候補日、調整URL、会員名
- 出力：送信用文面（候補日あり/なし）
- 分岐：営業投稿に候補日明示がない場合は「なし」扱い

### Job-B: 3日前/当日リマインド生成

- 入力：面談日、フォーム提出状況、未提出項目
- 出力：3日前依頼文、当日2-3時間前催促文
- 例外：未返信3日 + 前日未確定は電話フォロー指示を付与

### Job-C: 面談後フォロー文生成

- 入力：公開目標日、タスク期限、初週目安、月次見通し
- 出力：会員向けLINE草案
- チェック：必須5要素が本文に存在すること

### Job-D: 公開処理チェック補助

- 入力：決済確認状況、書類回収状況、設定値
- 出力：公開可否（Yes/No）と不足項目一覧

### Job-E: 内部共有文生成

- 入力：面談メモ、録画URL、Next Action
- 出力：Slack投稿文（今日話したこと/目標/面談メモ/Next Action）

## 5.2 品質ゲート（全ジョブ共通）

1. 宛先・会員名一致確認
2. 日時・URL整合確認
3. 文体確認（丁寧・絵文字なし）
4. 個人情報最小化
5. 送信前に人間承認

## 6. 日次1時間運用への落とし込み

## 6.1 17:00-18:00の標準配分

1. 17:00-17:10  
   Slackメンション確認、緊急度判定
2. 17:10-17:35  
   Claude Codeで返信草案・リマインド草案を生成
3. 17:35-17:50  
   人間レビューして送信/共有
4. 17:50-18:00  
   会員DB更新、翌日タスク確定

## 6.2 稼働外の扱い

- 原則は能動チェックしない
- 急ぎはSlackメンション起点のみ対応
- 稼働外対応は翌日ログへ必ず記録

## 7. データ設計（最低限）

## 7.1 スプレッドシート列（推奨）

- 会員ID
- 担当CS
- 公開目標日
- 公開実施日
- 面談予定日
- フォーム提出状況
- 未提出項目
- 最終連絡日
- 電話フォロー回数
- エスカレーション有無

## 7.2 KPI（週次）

- 初回連絡SLA遵守率
- 面談前フォーム提出率
- 公開目標日遵守率（目標日 vs 実績日）
- 未返信案件の滞留日数
- 1時間枠内完了率（17:00-18:00で完了した割合）

## 8. 実装ロードマップ（4週間）

### Week1

- テンプレ台帳整備
- NotebookLMソース投入
- Job-A/Job-Bの手動運用開始

### Week2

- Job-C/Job-E追加
- 品質ゲートチェックリスト運用開始

### Week3

- Job-D（公開前チェック）導入
- 週次定例の事前共有文自動下書き導入

### Week4

- KPI計測
- ボトルネック分析
- v2改善（文面精度、分岐精度、作業時間短縮）

## 9. まず明日着手する3つ

1. NotebookLMにRingwell用ソースを投入して質問テンプレを作る
2. Claude Codeで「日程調整文面生成」と「3日前催促文生成」を作る
3. 公開前チェックのYes/Noシートをスプレッドシートで作る

## 10. Job実装仕様（I/Oスキーマ）

この章は、誰が実装しても同じ挙動に揃えるための最小仕様。

### Job-A 日程調整メッセージ生成

- 必須入力
  - `member_name`（文字列）
  - `has_candidate_dates`（真偽値）
- 条件付き入力
  - `candidate_dates`（配列、`has_candidate_dates=true`のとき必須）
  - `scheduling_url`（文字列、`has_candidate_dates=false`のとき必須）
- 出力
  - `message_draft`（文字列）
  - `branch_type`（`candidate` or `url`）
  - `quality_flags`（配列）
- エラー返却
  - `missing_required_field`
  - `invalid_branch_data`

### Job-B 3日前/当日リマインド生成

- 必須入力
  - `member_name`
  - `meeting_datetime`
  - `form_status`（`submitted`/`not_submitted`）
- 任意入力
  - `missing_items`（配列）
  - `days_no_response`（数値）
- 出力
  - `remind_3days_draft`
  - `remind_same_day_draft`
  - `phone_followup_required`（真偽値）
- エラー返却
  - `invalid_meeting_datetime`

### Job-C 面談後フォロー文生成

- 必須入力
  - `member_name`
  - `publish_target_date`
  - `near_term_tasks`（配列）
  - `first_week_kpi`（配列）
- 任意入力
  - `monthly_outlook`（配列）
- 出力
  - `followup_message_draft`
  - `contains_required_elements`（真偽値）
  - `missing_elements`（配列）

### Job-D 公開前チェック補助

- 必須入力
  - `payment_verified_slack`（真偽値）
  - `payment_verified_mail`（真偽値）
  - `document_status_ok`（真偽値）
  - `profile_settings_ok`（真偽値）
- 出力
  - `publish_ready`（真偽値）
  - `missing_checks`（配列）
  - `next_actions`（配列）

### Job-E 内部共有文生成

- 必須入力
  - `today_topics`（配列）
  - `meeting_goal`（文字列）
  - `meeting_notes`（配列）
  - `next_actions`（配列）
- 任意入力
  - `recording_url`（文字列）
- 出力
  - `slack_post_draft`（文字列）
  - `format_valid`（真偽値）

## 11. 例外処理・運用ルール

### 11.1 システム例外

- API失敗時
  - 3回リトライ後に停止
  - `manual_fallback_required=true`を返す
- 入力不足時
  - 不足項目を明示して生成中止
- 重複送信防止
  - 同一会員/同一目的の送信は最終送信時刻から30分以内はブロック

### 11.2 業務例外

- 夜間/休日通知
  - 24時間以内返信ルールを適用
- 未返信継続
  - 3日後再催促 -> 電話2回 -> 統括/営業へエスカレーション
- 当日キャンセル
  - LINE連絡必須、再調整リンク再送、内部スレッド記録

## 12. RACI（責任分界）

- **実行責任（R）**：担当コンシェルジュ
- **最終承認（A）**：担当コンシェルジュ（重要案件は統括）
- **相談先（C）**：コンシェルジュ統括、営業担当、事務担当
- **共有先（I）**：関係Slackチャンネル、事務担当リーダー

業務別の最終承認ルール:

- 会員向け送信文：担当コンシェルジュ承認必須
- 公開処理可否：担当コンシェルジュ判断、決済未確認時は事務担当へ確認
- 難案件返信：必要に応じて統括レビュー

## 13. KPI運用手順（誰がいつ更新するか）

### 13.1 更新責任

- 日次更新（17:55-18:00）：担当コンシェルジュ
- 週次集計（月曜定例前）：担当コンシェルジュ
- 週次レビュー（月曜18:00定例）：統括主導

### 13.2 計測方法

- 初回連絡SLA遵守率
  - 分子：SLA内送信件数
  - 分母：当週新規アサイン件数
- フォーム提出率
  - 分子：面談前提出済み件数
  - 分母：当週面談件数
- 公開目標日遵守率
  - 目標日 <= 実績日の一致率
- 1時間枠内完了率
  - 17:00-18:00で完了したタスク件数 / 当日総タスク件数

### 13.3 KPI悪化時の対応

- 連続2週で悪化した指標は改善アクションを必須化
- 改善アクションは「担当」「期限」「成功条件」を設定

## 14. セキュリティ・個人情報ルール（実装必須）

- 生成プロンプトに個人情報を過剰投入しない
- 相談ログは匿名化してNotebookLMへ投入
- 録画URLはSlack投稿時にアクセス権を確認
- 外部共有禁止情報はテンプレに明記
- 誤送信時は
  1. 即時連絡停止
  2. 統括報告
  3. 影響範囲確認
  4. 再発防止メモ作成

## 15. 実装受け入れ基準（Definition of Done）

以下を満たしたら運用開始可。

1. Job-A〜Eが手動テストケースで80%以上通過
2. 送信前品質ゲートが全ジョブで有効
3. 日次1時間運用で1週間連続運用できる
4. KPIの更新と週次レビューが実施されている
5. 例外時の手動フォールバックが機能している

## 16. Studio/Opal導入版アーキテクチャ

Google Workspace中心業務は、Claude Code単体より `Studio/Opal + Claude Code` のハイブリッド構成が運用しやすい。

### 16.1 使い分け方針

- Claude Code
  - 分岐ロジック設計
  - 文面生成ロジック
  - 例外処理設計
- Google Workspace Studio / Opal
  - Gmail/Calendar/Sheets の実運用接続
  - 権限/共有/監査が必要なワークフロー運用
  - 非エンジニアでも触れる定型オペレーションの実行

### 16.2 Job実装分担（推奨）

- Job-A（日程調整文面）
  - 生成ロジック：Claude Code
  - 送信運用：Opal（またはStudio）
- Job-B（3日前/当日催促）
  - トリガー：Calendar/SheetベースでStudio/Opal
  - 文面生成：Claude Code
- Job-C（面談後フォロー）
  - 文面生成：Claude Code
  - 配布運用：Opal
- Job-D（公開前チェック）
  - データ収集：Studio/Opal（Slack/Gmail/Sheets）
  - 判定ロジック：Claude Code
- Job-E（内部共有文）
  - 整形ロジック：Claude Code
  - 投稿運用：Opal

### 16.3 導入順序（現実的）

1. まずClaude CodeでJob-A/Bのロジックを固める
2. 次にOpalで「送信・記録」部分を置き換える
3. 最後にStudioで監査・権限・承認を整備する

## 17. 全ツール編成（Typeless / Notta / Grok含む）

### 17.1 追加ツールの役割

- Typeless
  - 面談後の気づき、返信草案の音声メモ入力
  - 17:00開始前の思考ダンプ
- Notta
  - 面談音声/会議の安定した書き起こし取得
  - Geminiの補完として利用
- Grok（grock）
  - リアルタイム情報収集
  - 婚活市場トレンドや話題の外部確認

### 17.2 利用判断ルール

- Typelessを使う場面
  - タイピングより速くメモしたい時
  - 1時間枠内で思考整理コストを下げたい時
- Nottaを使う場面
  - 会議書き起こしの安定性を優先する時
  - 録音データを定型的に蓄積したい時
- Grokを使う場面
  - 外部トレンドや速報性が必要な論点
  - NotebookLMだけでは答えられない外部情報の補完

### 17.3 統合ワークフロー（拡張版）

1. 入力
   - Typeless（音声メモ）
   - Notta/Gemini（面談ログ）
   - Grok（外部情報）
2. 理解
   - NotebookLM（ソース限定検証）
3. 実行
   - Claude Code（分岐・文面・判定）
   - Studio/Opal（実運用連携）
4. 仕上げ
   - Cursor（最終統合）
5. 蓄積
   - Obsidian（再利用資産）

### 18.2 17:00-18:00

1. 17:00-17:05

## 19. テストケース（Job別）

本章は実装完了後の受け入れテストにそのまま利用する。

### 19.1 Job-A（日程調整メッセージ生成）

- 正常系1：候補日あり
  - 入力：候補日3件 + 会員名
  - 期待：`branch_type=candidate`、候補日を含む文面
- 正常系2：候補日なし
  - 入力：調整URL + 会員名
  - 期待：`branch_type=url`、URL入力依頼文面
- 正常系3：営業投稿に候補日未記載
  - 入力：`has_candidate_dates=false`
  - 期待：自動でURL分岐
- 異常系1：候補日ありだが候補日配列が空
  - 期待：`invalid_branch_data`
- 異常系2：会員名未入力
  - 期待：`missing_required_field`

### 19.2 Job-B（3日前/当日リマインド生成）

- 正常系1：3日前・未提出
  - 期待：3日前依頼文が生成される
- 正常系2：当日2時間前・未提出
  - 期待：当日催促文が生成される
- 正常系3：提出済み
  - 期待：催促文ではなく確認済み扱い
- 異常系1：面談日時フォーマット不正
  - 期待：`invalid_meeting_datetime`
- 異常系2：`form_status`不正値
  - 期待：入力エラーで停止

### 19.3 Job-C（面談後フォロー文生成）

- 正常系1：公開目標日/タスク/KPI入力あり
  - 期待：必須5要素を含む文面
- 正常系2：月次見通し未入力
  - 期待：他要素で文面生成可能
- 正常系3：タスク複数件
  - 期待：時系列で整形
- 異常系1：公開目標日なし
  - 期待：`missing_elements`に目標日
- 異常系2：タスク空
  - 期待：`contains_required_elements=false`

### 19.4 Job-D（公開前チェック補助）

- 正常系1：決済確認/書類/設定すべてOK
  - 期待：`publish_ready=true`
- 正常系2：Slack確認不可だがGmail確認OK
  - 期待：公開可
- 正常系3：設定値不足1件
  - 期待：不足項目のみ返却
- 異常系1：決済確認が双方NG
  - 期待：公開不可
- 異常系2：入力欠損
  - 期待：不足項目を明示

### 19.5 Job-E（内部共有文生成）

- 正常系1：全入力あり
  - 期待：固定4項目が順序どおり出力
- 正常系2：録画URLなし
  - 期待：投稿文は生成可能
- 正常系3：Next Action複数
  - 期待：箇条書き整形
- 異常系1：目標未入力
  - 期待：`format_valid=false`
- 異常系2：面談メモ空
  - 期待：入力不足エラー
