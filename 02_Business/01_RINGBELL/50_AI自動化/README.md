# RINGBELL AI自動化 — フォルダガイド＆設計原則

## なぜこのフォルダを作ったか

- **目的:** AI導入・業務改善コンサルを、ドメイン非依存の同じ型で回せるようにする。RINGBELL（婚活CS）の実装と、再利用可能な汎用テンプレを分離する。
- **狙い:** スコープ・As-Is/To-Be・説明責任・HITL・リスク/経済性・定着・資産化まで一連の納品品質を、テンプレとチェックリストで再現可能にする。

## この棚の約束（プロ視点 × 初心者OK）

1. **業務と契約が先** — 何を直すか・やらないか・誰が説明するかを固定してから、AIを当てはめる（AIありきにしない）
2. **小さく回して測る** — 指標・PoC・撤退条件を先に置き、拡大は勝ってから
3. **人が最終確定する場所** — HITL（承認・例外・顧客向け最終稿）は必ず明示する
4. **1か所1役** — 同じ事実を2ファイルに「正」として書かない

**簡略化したのは同じ内容の二重記述のみ。** 意思決定に効く論点の幅（スコープ/As-Is・To-Be/責任分界/リスク・規約/経済性・撤退/運用・KPI/監査可能さ）は、ロードマップ・施策一覧・Executive チェックリストに**削らず**置いてある。

## 正本ルール（重複を避ける）

**同じ判断材料は、更新するときに触るファイルを1つに絞る。**

| 情報の種類 | 正本ファイル | 他での扱い |
|:--|:--|:--|
| **全体進捗・いま何番までか** | `統合実行計画_Phase1-3.md` の Phase/Week | 他ファイルに「全体進捗」を書かない |
| **週次のタスク・厚み** | `30_実行計画/テンプレ_実行計画サマリー.md` の複製 | 実行計画は通し番号とリンク用に留める |
| **施策の存在・ステータス** | `40_施策実装/00_施策一覧.md` + 各設計書 | 一覧=レジストリ、設計書=詳細 |
| **テンプレのファイル一覧** | **本ファイル（README.md）のフォルダツリ** | 各 `10`～`50` の `00_README.md` は役割1-2行+本ファイル参照のみ |

## ドメイン非依存に保つ（汎用性）

- **本フォルダ内の `テンプレ_*` には、業界名・社名・固有システム名を書き込まない。** 案件固有はすべて複製したファイル側に書く
- **追加論点** — `10_業務整理/テンプレ_ヒアリング項目一覧.md` の「追記」セクションか、複製にだけ足す（正本の表は薄く保つ）
- **大企業・規制案件** — 本棚は個人事業〜中小の実行可能な最小十分。DD・法務は顧客プロセスのテンプレを正とし、合意事項だけ `30_実行計画/テンプレ_リスクと外部サービス.md` に要約

## 拡張ルール（壊さず増やす）

| やりたいこと | 推奨パターン |
|:--|:--|
| **新しい施策が増える** | `40_施策実装/` に **1ファイル1施策** で追加し、`00_施策一覧.md` に1行追加 |
| **新フェーズが増える** | `60_` など**次の番号フォルダ**を1つ追加し、`統合実行計画_Phase1-3.md` に Phase を足す（`Phase2` 等の平行フォルダは作らない） |
| **既存フェーズ内に型だけ足す** | 同じ `NN_` 下に `テンプレ_新しい用途.md` を追加し、本ファイル（README）のツリと `統合実行計画` のステップを更新 |

**型の改版の流れ:** 変更は**必ず汎用テンプレ正本（`04_AI/04_テンプレート/`）を先**に直す → 作業棚へ必要分だけ反映
