## 0. 目的

`../../10_業務基盤/10_CS業務フロー分解.md`（旧 `Ringwell_CS_業務分解_初版.md`）を実行可能な自動化設計へ変換する。
目標は、CS品質を落とさずに「毎日17:00-18:00の1時間運用」で回る業務構造を作ること。

## 1. 設計原則

1. 人間は判断、AIは処理を担当する
2. 1ツール1責任（役割を混ぜない）
3. 返信前は必ず品質ゲートを通す
4. 例外時は手動へフォールバックする
5. ログを残し、毎週改善する

## 2. ツール責務マップ

### NotebookLM（理解・検証レイヤー）

- 公式テンプレ、運用ルール、FAQ、過去対応ログをソース化
- ケース別に「どう返すべきか」を根拠付きで整理
- 定例前の難案件の論点整理

### Claude Code（実行・自動化レイヤー）

- Slack通知/フォーム入力をトリガーに文面草案を生成
- 分岐ルールに沿って日程調整・リマインド文を自動生成
- 公開前チェックリストの自動判定補助
- Slack内部共有文（面談メモ/Next Action）の整形

### Cursor（編集・統合レイヤー）

- 文面最終化、運用ルール更新、テンプレ管理
- 複数ケース横断の改善反映

### 補助ツール

- ChatGPT：PR文生成メーカー、返信草案補助
- Gemini：面談ログ要約
- Google Workspace（Calendar/Gmail/Sheet）：実行トリガーと業務データの正本
- Slack：運用通知・エスカレーション・内部共有

## 3. 対象業務と自動化優先順位

## Phase 1（即効性重視）

1. 日程調整文面生成（候補日あり/なし）
2. 3日前フォーム依頼 + 未提出再催促文生成
3. 面談後フォロー文生成（公開目標日/締切/初週目安）

## Phase 2（公開処理の安定化）

4. 決済確認チェック補助（Slack/Gmail確認手順のガイド化）
5. 公開前Yes/Noチェックリスト生成
6. 公開完了連絡（会員向けLINE + 内部Slack投稿文）生成

## Phase 3（運用品質の底上げ）

7. 相談チャットの一次分類（定型回答/要判断/要エスカレーション）
8. 週次定例の事前共有メモ自動下書き
9. 面談メモからNext Action抽出・整形

## 4.3 AI生成物の品質基準（RINGBELL向け原則）

本棚でのAI生成は、すべて以下の**二者監修制**を前提とします：

| 場面 | AI生成物 | 最終責任者 | 確認項目 |
|:--|:--|:--|:--|
| **返信文生成** | Claude Code の草案文 | コンシェルジュ | 敬語・トーン・個人情報マスク・URLリンク |
| **面談ログ整形** | Claude Code の要約・Next Action | コンシェルジュ ± NotebookLM | 解釈のずれ・固有情報の残存・テンプレ整合 |
| **テックタッチ動画台本** | ChatGPT/Gemini の初稿 | コンシェルジュ | CTA・文体・エビデンス・動画の企画意図 |
| **FAQ・テンプレ化** | Claude Code の資産化候補 | オーナー | 匿名化・再利用可否・他施策との重複 |

**「生成 → 承認 → 送信」の3段階を徹底**し、「AI出力 = 最終形」と見なさない。

## 5. Claude Code自動化設計

## 5.1 自動化単位（ジョブ）

### Job-A: 日程調整メッセージ生成

- 入力：候補日有無、候補日、調整URL、会員名
- 出力：送信用文面（候補日あり/なし）
- 分岐：営業投稿に候補日明示がない場合は「なし」扱い

### Job-B: 3日前/当日リマインド生成

- 入力：面談日、フォーム提出状況、未提出項目
- 出力：3日前依頼文、当日2-3時間前催促文
- 例外：未返信3日 + 前日未確定は電話フォロー指示を付与

### Job-C: 面談後フォロー文生成

- 入力：公開目標日、タスク期限、初週目安、月次見通し
- 出力：会員向けLINE草案
- チェック：必須5要素が本文に存在すること

### Job-D: 公開処理チェック補助

- 入力：決済確認状況、書類回収状況、設定値
- 出力：公開可否（Yes/No）と不足項目一覧

### Job-E: 内部共有文生成

- 入力：面談メモ、録画URL、Next Action
- 出力：Slack投稿文（今日話したこと/目標/面談メモ/Next Action）

## 5.2 品質ゲート（全ジョブ共通）

1. 宛先・会員名一致確認
2. 日時・URL整合確認
3. 文体確認（丁寧・絵文字なし）
4. 個人情報最小化
5. 送信前に人間承認

## 6. 日次1時間運用への落とし込み

## 6.1 17:00-18:00の標準配分

1. 17:00-17:10  
   Slackメンション確認、緊急度判定
2. 17:10-17:35  
   Claude Codeで返信草案・リマインド草案を生成
3. 17:35-17:50  
   人間レビューして送信/共有
4. 17:50-18:00  
   会員DB更新、翌日タスク確定

## 6.2 稼働外の扱い

- 原則は能動チェックしない
- 急ぎはSlackメンション起点のみ対応
- 稼働外対応は翌日ログへ必ず記録

## 10. Job実装仕様（I/Oスキーマ）

この章は、誰が実装しても同じ挙動に揃えるための最小仕様。

### Job-A 日程調整メッセージ生成

- 必須入力
  - `member_name`（文字列）
  - `has_candidate_dates`（真偽値）
- 条件付き入力
  - `candidate_dates`（配列、`has_candidate_dates=true`のとき必須）
  - `scheduling_url`（文字列、`has_candidate_dates=false`のとき必須）
- 出力
  - `message_draft`（文字列）
  - `branch_type`（`candidate` or `url`）
  - `quality_flags`（配列）
- エラー返却
  - `missing_required_field`
  - `invalid_branch_data`

### Job-B 3日前/当日リマインド生成

- 必須入力
  - `member_name`
  - `meeting_datetime`
  - `form_status`（`submitted`/`not_submitted`）
- 任意入力
  - `missing_items`（配列）
  - `days_no_response`（数値）
- 出力
  - `remind_3days_draft`
  - `remind_same_day_draft`
  - `phone_followup_required`（真偽値）
- エラー返却
  - `invalid_meeting_datetime`

### Job-C 面談後フォロー文生成

- 必須入力
  - `member_name`
  - `publish_target_date`
  - `near_term_tasks`（配列）
  - `first_week_kpi`（配列）
- 任意入力
  - `monthly_outlook`（配列）
- 出力
  - `followup_message_draft`
  - `contains_required_elements`（真偽値）
  - `missing_elements`（配列）

### Job-D 公開前チェック補助

- 必須入力
  - `payment_verified_slack`（真偽値）
  - `payment_verified_mail`（真偽値）
  - `document_status_ok`（真偽値）
  - `profile_settings_ok`（真偽値）
- 出力
  - `publish_ready`（真偽値）
  - `missing_checks`（配列）
  - `next_actions`（配列）

### Job-E 内部共有文生成

- 必須入力
  - `today_topics`（配列）
  - `meeting_goal`（文字列）
  - `meeting_notes`（配列）
  - `next_actions`（配列）
- 任意入力
  - `recording_url`（文字列）
- 出力
  - `slack_post_draft`（文字列）
  - `format_valid`（真偽値）

## 11. 例外処理・運用ルール

### 11.1 システム例外

- API失敗時
  - 3回リトライ後に停止
  - `manual_fallback_required=true`を返す
- 入力不足時
  - 不足項目を明示して生成中止
- 重複送信防止
  - 同一会員/同一目的の送信は最終送信時刻から30分以内はブロック

### 11.2 業務例外

- 夜間/休日通知
  - 24時間以内返信ルールを適用
- 未返信継続
  - 3日後再催促 -> 電話2回 -> 統括/営業へエスカレーション
- 当日キャンセル
  - LINE連絡必須、再調整リンク再送、内部スレッド記録

## 12. RACI（責任分界）

- **実行責任（R）**：担当コンシェルジュ
- **最終承認（A）**：担当コンシェルジュ（重要案件は統括）
- **相談先（C）**：コンシェルジュ統括、営業担当、事務担当
- **共有先（I）**：関係Slackチャンネル、事務担当リーダー

業務別の最終承認ルール:

- 会員向け送信文：担当コンシェルジュ承認必須
- 公開処理可否：担当コンシェルジュ判断、決済未確認時は事務担当へ確認
- 難案件返信：必要に応じて統括レビュー

## 13. KPI運用手順（誰がいつ更新するか）

### 13.1 更新責任

- 日次更新（17:55-18:00）：担当コンシェルジュ
- 週次集計（月曜定例前）：担当コンシェルジュ
- 週次レビュー（月曜18:00定例）：統括主導

### 13.2 計測方法

- 初回連絡SLA遵守率
  - 分子：SLA内送信件数
  - 分母：当週新規アサイン件数
- フォーム提出率
  - 分子：面談前提出済み件数
  - 分母：当週面談件数
- 公開目標日遵守率
  - 目標日 <= 実績日の一致率
- 1時間枠内完了率
  - 17:00-18:00で完了したタスク件数 / 当日総タスク件数

### 13.3 KPI悪化時の対応

- 連続2週で悪化した指標は改善アクションを必須化
- 改善アクションは「担当」「期限」「成功条件」を設定

   - Claude Code（分岐・文面・判定）

4. 仕上げ
   - Cursor（最終統合）
5. 蓄積

## 19. テストケース（Job別）

本章は実装完了後の受け入れテストにそのまま利用する。

### 19.1 Job-A（日程調整メッセージ生成）

- 正常系1：候補日あり
  - 入力：候補日3件 + 会員名
  - 期待：`branch_type=candidate`、候補日を含む文面
- 正常系2：候補日なし
  - 入力：調整URL + 会員名
  - 期待：`branch_type=url`、URL入力依頼文面
- 正常系3：営業投稿に候補日未記載
  - 入力：`has_candidate_dates=false`
  - 期待：自動でURL分岐
- 異常系1：候補日ありだが候補日配列が空
  - 期待：`invalid_branch_data`
- 異常系2：会員名未入力
  - 期待：`missing_required_field`

### 19.2 Job-B（3日前/当日リマインド生成）

- 正常系1：3日前・未提出
  - 期待：3日前依頼文が生成される
- 正常系2：当日2時間前・未提出
  - 期待：当日催促文が生成される
- 正常系3：提出済み
  - 期待：催促文ではなく確認済み扱い
- 異常系1：面談日時フォーマット不正
  - 期待：`invalid_meeting_datetime`
- 異常系2：`form_status`不正値
  - 期待：入力エラーで停止

### 19.3 Job-C（面談後フォロー文生成）

- 正常系1：公開目標日/タスク/KPI入力あり
  - 期待：必須5要素を含む文面
- 正常系2：月次見通し未入力
  - 期待：他要素で文面生成可能
- 正常系3：タスク複数件
  - 期待：時系列で整形
- 異常系1：公開目標日なし
  - 期待：`missing_elements`に目標日
- 異常系2：タスク空
  - 期待：`contains_required_elements=false`

### 19.4 Job-D（公開前チェック補助）

- 正常系1：決済確認/書類/設定すべてOK
  - 期待：`publish_ready=true`
- 正常系2：Slack確認不可だがGmail確認OK
  - 期待：公開可
- 正常系3：設定値不足1件
  - 期待：不足項目のみ返却
- 異常系1：決済確認が双方NG
  - 期待：公開不可
- 異常系2：入力欠損
  - 期待：不足項目を明示

### 19.5 Job-E（内部共有文生成）

- 正常系1：全入力あり
  - 期待：固定4項目が順序どおり出力
- 正常系2：録画URLなし
  - 期待：投稿文は生成可能
- 正常系3：Next Action複数
  - 期待：箇条書き整形
- 異常系1：目標未入力
  - 期待：`format_valid=false`
- 異常系2：面談メモ空
  - 期待：入力不足エラー

## 23. 別枠: AI自動化以外のCS改善（同一ファイルで管理）

本章は「AI自動化」から外れるが、CS成果に効く施策を別枠で管理する。
運用の主軸は `20_設計/Ringwell_CS_AI自動化設計.md` に集約し、施策実行時は `40_施策実装/` に起票する。

### 23.1 CS改善のテックタッチ化

- 目的
  - 面談依存を下げ、節目での離脱・停滞を予防する
- 方針
  - 会員状態ではなく「イベント起点」で配信する
  - 正本チャネルはLINEとする
- 対象イベント（初期）
  1. 入会後1週間
  2. 入会後1ヶ月
  3. 入会後2ヶ月
  4. 休会復帰時
  5. 真剣交際ブレイク時（一次接触SLA: 72時間）
- 最低KPI
  - 配信到達率
  - 反応率（既読/クリック/返信のいずれか）
  - 配信後7日以内の行動再開率

### 23.2 動画化

- 目的
  - 同じ説明の繰り返しを減らし、会員の自己解決率を上げる
- 制作ルール
  - 1本あたり制作工数は最大90分
  - 1動画1テーマで、視聴後の次アクション（CTA）を明示する
- 初期制作対象（5本）
  1. 入会1週間向け（初期迷子防止）
  2. 入会1ヶ月向け（継続行動の立て直し）
  3. 入会2ヶ月向け（停滞予防・意思決定）
  4. 休会復帰向け（再起動）
  5. ブレイク向け（感情整理と再スタート）
- 最低KPI
  - 視聴率（または完了率）
  - 視聴後7日以内のアクション率
  - 問い合わせ重複率の低下

### 23.3 実装との接続ルール

- テックタッチ/動画施策は、`40_施策実装/00_施策一覧.md` にID登録する
- 施策ごとに `テンプレ_施策設計書.md` を複製し、PoC -> 本番の順で進める
- 会員向け公開前は必ず人間レビューを通す
