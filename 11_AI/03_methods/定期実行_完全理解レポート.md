# 定期実行 完全理解レポート【最終版】

---

## はじめに：このレポートで何がわかるか

Claude Codeを使った定期実行には4つの手段がある。
どれを選ぶかを間違えると「動かない」「止まった」「AIが判断してくれない」という事態になる。

このレポートを読めば：
- 4つの手段の本質的な違いが理解できる
- 自分の用途に合った手段をすぐ選べる
- 明日から実際に設定できる

**対象読者：** Claude Codeを使い始めた・使い始めようとしているすべての人

---

## 1. 定期実行とは何か

「決まった時間に自動でタスクを実行する仕組み」のこと。

一度設定すれば、人間が寝ている間・移動中・別の作業中でも処理が動き続ける。

**定期実行で自動化できる業務の例：**

```
毎朝7時   → ニュースをまとめてSlackに送る
毎週月曜  → ダウンロードフォルダを自動整理する
毎晩0時   → Salesforceのデータを集計してレポートを作る
毎日      → CIのステータスを確認して失敗があれば自動修正する
```

---

## 2. 選び方の基本：2軸で整理する

定期実行の手法は以下の2つの質問に答えるだけで選び方が決まる。

```
質問1：AIに判断させたいか？
質問2：PCが閉じていても動いてほしいか？
```

この2軸で4つに分類できる：

```
                  PC必要（ローカル）      PC不要（クラウド）
AIなし       →   cron                  GitHub Actions
AIあり       →   /loop                 /schedule ★本命
```

> **「AIに判断させたいか」「PCの電源状態に依存したくないか」**
> この2点を先に自分に問いかけてから手法を選ぶと、迷いが大幅に減る。

## 3. 前提知識：cron式の読み方

/scheduleやGitHub Actionsを使うとき「cron式」という時刻指定の書き方が出てくる。
難しく見えるが、構造を知れば5分で読めるようになる。

### cron式の構造

```
┌─────── 分（0-59）
│ ┌───── 時（0-23）
│ │ ┌─── 日（1-31）
│ │ │ ┌─ 月（1-12）
│ │ │ │ ┌ 曜日（0-7、0と7が日曜）
│ │ │ │ │
* * * * *
```

`*`（アスタリスク）は「毎回」を意味する。

### よく使うcron式の例

```
0 7 * * *     → 毎日朝7時
0 9 * * 1     → 毎週月曜の朝9時
0 0 * * *     → 毎日深夜0時
0 7 * * 1-5   → 平日（月〜金）の朝7時
0 */6 * * *   → 6時間おき
```

### 覚え方のコツ

最初の2つ（分・時）だけ覚えればほぼ対応できる。
残りの3つは基本的に`*`（毎回）にしておけばOK。

## 4. 4つの手段を徹底解説

---

### ① cron（クーロン）── タイマー単体

**一言で言うと「目覚まし時計」。AIは一切入っていない。**

#### cronの本質

cronは時計係。「この時間になったらこれを起動して」と指示するだけで、
中身を何も理解しない。「読む」「判断する」「修正する」という概念がそもそも存在しない。

```
【cronができること】
  ✓ 決まった時間にプログラムを起動する
  ✓ シンプルで信頼性が高い
  ✓ 有効期限がなく無期限で動き続ける

【cronができないこと】
  ✗ 状況を判断する
  ✗ エラーの原因を分析する
  ✗ 内容を変えて実行する
```

#### cron単体で十分なケース

```
毎日0時にバックアップを取る
毎週月曜にログファイルを削除する
→「毎回同じことをするだけ」の定型処理はcron単体で十分
```

#### cron + Claude Codeの組み合わせ

「cronでAI自動化してます」という話は、
**cronが起動した先にClaude Codeが動いているという構造**のこと。
AIを担当しているのはClaude Codeであって、cronはただのトリガーにすぎない。

```
【通常のcronジョブ】
  cronが起動 → 固定コードを毎回同じように実行するだけ

【cron + Claude Codeの組み合わせ】
  ① cronが「7:00になった、起動しろ」と命令
  ② Claude Codeが起動してプロンプトを読む
  ③ データを集計・分析・文章化
  ④ MCP（Slack等）経由で結果を送信
  → 「毎回同じように実行する」から「毎回考えて実行する」に変わる
```

**Claude Codeが加わることで何が変わるか：**

```
・データの変化に応じてコメントや警告のトーンを変える
・エラーの原因を読んで修正コードを書く
・プロンプトを更新するだけで挙動が変わる（コードを書き直す必要がない）
```

#### cronとClaude Codeの接続方法

```bash
# crontabを編集（ターミナルで実行）
crontab -e

# 以下を追記（毎朝7時にClaude Codeを起動する例）
0 7 * * * cd ~/work/project && claude --print "今日のSalesforceデータを集計してSlackに送って" 
```

**ポイント：**
- `--print` オプションでClaude Codeを非対話モードで起動できる
- プロンプトの内容を変えるだけで実行内容を変更できる
- PCが起動していないと動かない点に注意

#### 注意点

- PCが起動していないと動かない
- cronにAIは入っていない。この誤解が最も多いパターン
- 「cronにAIが入っている」と思って使い始めると必ず詰まる

### ② /loop（ループ）── セッション内監視

**一言で言うと「その場限りのAIタイマー」。**

Claude Codeのセッション内だけで動く一時的なスケジューラ。

#### 使い方

```bash
# Claude Codeのターミナル内で入力するだけ
/loop 5m CIのステータスを確認して、失敗していたら内容をまとめて

# 書き方
/loop [間隔] [指示内容]

# 間隔の指定例
/loop 5m  → 5分ごと
/loop 1h  → 1時間ごと
/loop 30s → 30秒ごと
```

#### cronとの最大の違い

**自然言語で指示できる点**。cron式を書く必要がなく、
指示内容の調整もその場でできる。

#### 向いているケース

```
・デプロイ中の進捗を30分だけ監視したい
・PRのレビューが完了したか1時間おきに確認したい
・セッション中の繰り返し作業に使いたい
```

#### 致命的な制約

```
・ターミナルを閉じた瞬間に止まる　← 最重要
・セッション内最大50タスクまで
・「今だけ動けばいい」という一時的な用途専用
・翌日も動かし続けたい処理には絶対に向かない
```

### ③ /schedule（スケジュール）── クラウドエージェント★本命

**一言で言うと「PCを閉じても動くAIタイマー」。**

2026年3月24日にAnthropicが発表した機能。
「Mac mini要らないじゃん」という言葉が話題になったのはここから。

以前はAIを常時動かすためにPCを24時間つけっぱなしにするか、
サーバーを借りるかMac miniを常時稼働させるしかなかった。
それがコマンド1行で解決するという話なので、反響が大きかった。

#### 設定手順（コピペ可能）

**Step 1：Claude Codeのターミナルで /schedule と入力**

```bash
/schedule
```

**Step 2：リポジトリを指定**

```
Which repository should this run against?
→ owner/repository-name を入力
例）masahiro/work-automation
```

**Step 3：スケジュールをcron式で指定**

```
When should this run? (cron expression)
→ cron式を入力
例）0 7 * * *（毎朝7時）
```

**Step 4：プロンプトを自然言語で入力**

```
What should Claude do?
→ 日本語でそのまま入力できる
例）今日のSlackの未読メッセージを要約して、重要なものをリストアップしてください
```

**Step 5：確認して完了**

```
→ Anthropicのクラウドが引き受けてくれる
→ PCを閉じても毎朝7時に自動実行される
```

#### 実際の活用例（Anthropic社内）

```
・CIの失敗を自動検出 → 修正コードを書いてコミット
・ドキュメントの定期更新とプッシュ
・パッチレベルの依存パッケージを毎週月曜に自動アップグレード → PR提出
・2日以上放置されているPRにフラグを立てる
```

#### 職種別・業務別の活用シナリオ

```
【営業職】
  毎朝8時 → Salesforceの昨日の活動を集計して
             今日フォローすべきリードをリストアップしてSlackに送る

【マーケター】
  毎朝9時 → GA4のデータを取得して
             昨日のトラフィック異常を検知・レポートをNotionに保存

【SaaS CS担当】
  毎日18時 → 当日チャーンリスクの高い顧客を特定して
              CSチームのSlackチャンネルに通知する

【経営者・管理職】
  毎週月曜9時 → 先週の売上・KPI・課題を集計して
                週次ブリーフィングをGmailの下書きに作成する

【コンテンツ担当】
  毎週日曜22時 → 今週のAIニュースをX・メルマガ・各種ソースから収集して
                 月曜朝に配信する記事の下書きをNotionに作成する
```

#### コスト感覚

```
/scheduleのコスト ＝ 実行時に使ったトークン数 × 通常料金

目安：
  シンプルな集計レポート（毎朝）  → 約1,000〜5,000トークン/回
  複数ソースの調査・記事作成      → 約10,000〜50,000トークン/回
  Agent Teams使用時              → 約100,000トークン以上/回

Claude Pro（$20/月）の場合：
  シンプルなタスクなら毎日実行しても
  通常の利用制限内に収まることが多い
  ただしAgent Teamsや大量データ処理は消費が激しいため注意
```

#### 重要な注意点：3日で自動失効

```
/scheduleは作成から3日で自動失効する。

これはバグではなくAnthropicによる意図的な安全装置。
理由：
  ・設定したまま忘れて暴走するリスクを防ぐ
  ・意図しないコスト超過を防ぐ

対策：
  ・カレンダーに「2日後に再設定」のリマインダーを入れる
  ・長期運用ならGitHub Actionsへ移行する
```

#### cronと/scheduleの本質的な違い

```
cron      → 「定時起動装置」
             CIが失敗しても「失敗した」と記録するだけ
             なぜ失敗したか・どう直すかは人間が見に行く必要がある

/schedule → 「定時エージェント実行装置」
             Claudeが失敗ログを読んで原因特定
             → 修正コードを書いてコミットまで完結
```

**具体例：AIニュースまとめを毎週自動作成する場合**

```
【cronの場合】
  毎週日曜夜に起動
  → 起動するだけ。情報収集も記事化も人間の仕事

【/scheduleの場合】
  毎週日曜夜に起動
  → ClaudeがX投稿・海外メルマガ・ネットの情報を自動収集
  → 何が重要かを判断して記事化
  → ファイル保存まで完了
  → 人間は月曜朝に完成した記事を確認するだけ
```

### ④ GitHub Actions + Claude Code ── チームの最終解

**一言で言うと「/scheduleの無期限版・チーム共有版」。**

#### 前提条件（初心者向け）

```
必要なもの：
  ✓ GitHubアカウント（無料で作成可能）
  ✓ GitHubリポジトリ（プロジェクトの保存場所）
  ✓ Anthropic APIキー

難易度：
  ★★★☆☆（/scheduleより設定が複雑）
  エンジニアがいるチームなら任せるのが現実的
  個人でも学習コスト1〜2時間あれば設定可能
```

#### /scheduleとの違い

```
/schedule              GitHub Actions
──────────────────────────────────────
3日で失効        →    無期限稼働
個人のみ         →    チームで設定を共有できる
設定が簡単       →    YAML設定が必要（やや複雑）
リポジトリ必須   →    リポジトリ必須
```

#### GitHub Actionsを選ぶべき具体的な条件

以下のひとつでも当てはまるならGitHub Actionsが自然な選択：

```
① チームで共有したい
   cronはローカル設定で個人依存になりやすい
   /scheduleの3日失効は管理が煩雑
   GitHub Actionsはリポジトリにコードとして定義されるため
   バージョン管理もできてチーム全員が設定を把握できる
   → 誰かが退職・異動しても設定が消えることがない

② /scheduleの3日失効が運用の邪魔になってきた
   長期稼働が必要なタスクを3日ごとに再設定するのは持続しない
   → GitHub Actionsなら一度設定すれば無期限で動き続ける

③ 既存のCI/CDパイプラインに組み込みたい
   anthropics/claude-code-action@v1 を使うことで
   「無期限 × クラウド × AI判断」の3条件を同時に満たす
   現時点でこれが唯一の構成
```

#### 設定例（コピペ可能）

```yaml
# .github/workflows/daily-report.yml
# このファイルをGitHubリポジトリの .github/workflows/ に置くだけ

name: Daily AI Report
on:
  schedule:
    - cron: '0 7 * * *'  # 毎朝7時（UTC）に実行
                          # 日本時間は+9時間なので16時に相当
                          # 日本時間朝7時にしたい場合は '0 22 * * *' にする

jobs:
  run-claude:
    uses: anthropics/claude-code-action@v1
    with:
      prompt: "今日のSalesforceデータを集計してSlackに送って"
    secrets:
      anthropic-api-key: ${{ secrets.ANTHROPIC_API_KEY }}
```

**注意：GitHub ActionsのスケジュールはUTC（協定世界時）で動く。**
日本時間（JST）はUTC+9時間なので、設定時は9時間引いた時刻を入力する。

```
日本時間 朝7時 → UTC 22時（前日）→ cron式: 0 22 * * *
日本時間 朝9時 → UTC 0時         → cron式: 0 0 * * *
日本時間 18時  → UTC 9時         → cron式: 0 9 * * *
```

## 7. よくある失敗パターンと対策

| 失敗 | 原因 | 対策 |
|---|---|---|
| /loopでPCを閉じたらタスクが消えた | /loopはセッション限定の設計 | PC不要で動かしたいなら/scheduleを使う |
| /scheduleが3日後に突然止まった | 3日の有効期限が仕様 | カレンダーに「2日後に再設定」リマインダーを入れるかGitHub Actionsへ移行 |
| cronを使ったがCI失敗に気づかなかった | cronは実行するだけで判断・修正は人間の仕事 | AIに対処させたいならClaude Codeと組み合わせる |
| 「cronにAIが入っている」と思って使い始めた | cronにAIはない。AI部分を担うのはClaude Code | 設計思想を根本から変える必要がある |
| GitHub ActionsがUTC時刻で動いてズレた | GitHub ActionsはUTC基準で動く | 日本時間から9時間引いた時刻をcron式に入力する |

## 8. Hooksとの違い：定期実行ではないが混同しやすい

Hooksは「定期実行」ではなく「イベント駆動」の自動化。

```
定期実行（時間で起動）：
  cron / /loop / /schedule / GitHub Actions
  → 「毎朝7時になったら実行」

Hooks（イベントで起動）：
  → 「コマンドが実行される直前に自動チェック」
  → 「ファイルが保存されたら自動フォーマット」
  例）settings.jsonで設定 → deny-check.shで危険なコマンドを自動ブロック
```

定期実行と組み合わせることで「時間でも・イベントでも」自動化が完成する。

## 10. まとめ：3行で言うと

1. **2軸で選ぶ。** 「AIが必要か」「PC不要か」の2点を先に決めれば迷わない。
2. **個人で今すぐ試すなら/schedule。** PCを閉じても動く・設定1行・AI判断あり。ただし3日失効に注意。
3. **チームで永続運用するならGitHub Actions。** /scheduleの上位互換として、無期限・チーム共有・CI/CD統合が可能。
