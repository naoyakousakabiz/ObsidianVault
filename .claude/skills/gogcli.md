# /gogcli — Google Workspace（gog）

## これは何か

Gmail やカレンダー、Drive、Sheets を **gog（gogcli）** で触るときの**コマンドの組み立て方**と、安全に試す順序を出し、可能な範囲は **実際にコマンドを実行**して結果まで返すスキルです。削除や一括送信は**二重確認**を前提にします。

## いつ使うか

- 条件で検索したメールにラベルを付けたい、ファイルを移したいとき
- カレンダーやシートを**スクリプトの入出力**にしたいとき

## 使わないほうがよい場合

- 要約や一覧だけでよく、gogcli まで不要なとき（`morning-briefing` など）
- 認証や CLI がまだ整っていないとき（先に環境づくり）

## 前提条件

- gogcli が認証済みであること。**トークンや秘密を会話に貼らない**。
  - 目安: `gog gmail search ...` が成功する

## どう依頼するか

1. **`gogcli`** と、やりたいことを動詞で具体的に（例: 直近 7 日の未読にラベル X）。
2. **本番アカウントかテストか**をはっきり書く。
3. 失敗してよい範囲（ロールバックの有無）。

**例:** 「`gogcli`。テストアカウント。Drive の `Report` 以下の 2025 年 PDF を一覧し、Sheets に出す案が欲しい。」

## 進め方（エージェント向け）

1. **読む → 件数を見る → 書き換える**の順で段階化する。
2. 削除・送信・権限変更は、**影響件数**を先に示す。
3. 同じコマンドを二度叩いても壊れにくい、**冪等に近い**形を意識する。
4. API のクォータを踏まえ、分割や待機を提案する。

## 実行ポリシー（重要）

- ユーザーが「Gmailで…」「カレンダーで…」のように **gog の単語を出していなくても**、Google Workspace 操作の依頼なら **gog を第一候補**にしてよい（MCP の有無に引きずられない）。
- 変更系は **`--dry-run` → 実行**の二段階。実行前に「影響（件数・対象）」を短く提示する。
- `GOG_ACCOUNT` は、ユーザー指定があれば優先し、無ければ環境変数または既定を使う。コマンドには必要に応じて `-a <account>` を付ける。

## 完了の目安

- コマンドが**試す順序**がわかる並びになっている
- 危ない本番操作に**警告**がある
- 秘密がログに残らない書き方になっている
