# /simplify — コードの簡素化レビュー

## これは何か

変更箇所や指定ファイルを、**読みやすさ・名前・重複・境界条件**の観点で見て、**小さなパッチ単位**の改善案を返すスキルです。挙動を変える提案は、依頼がない限り**控える**。

## いつ使うか

- PR を出す前に、第三者目線の指摘が欲しいとき
- 動くけれど読みにくいコードを、**リスクを抑えて**薄くしたいとき

## 使わないほうがよい場合

- アーキテクチャ全体の設計変えが主眼のとき
- コードではなくドキュメントだけのレビューのとき

## 前提条件

- リポジトリを読めること。リンターやコーディング規約があればそれに合わせる。

## どう依頼するか

1. **`simplify`** とファイルパスまたは diff。
2. 優先したい軸（可読性、パフォーマンス、公開 API は触らない、など）。
3. 変更要望の上限があれば（例: この PR では 200 行まで）。

**例:** 「`simplify`。`src/foo.ts` のこの PR。公開関数のシグネチャは変えないで。」

## 進め方（エージェント向け）

1. 範囲を決め、既存のスタイルに合わせる。
2. 指摘は**きつさ付き**で列挙し、それぞれに小さな修正案を付ける。
3. 境界条件やテストの観点を短く書く。
4. 挙動が変わりそうな案は**別オプション**として分ける。

## 完了の目安

- どの行・どのパターンか、指摘がイメージできる
- 修正案が**小さく当てられる**単位になっている
- プロジェクトの規約と矛盾していない
