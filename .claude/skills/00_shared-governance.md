# スキル共通：安全・品質・運用（正）

各スキル `.md` の**重複を避ける**ため、共通前提だけをここに集約する。個別スキルは末尾で本ファイルを指す。

**Cursor 常時文脈:** Vault 根 `CLAUDE.md` の **「エージェントスキル（`.claude/skills`）」** に、本ファイルと**同趣旨の要約**を置く。機微・ガバナンスの**最終枠組み**は `04_AI/01_運用/02_フェーズとガバナンス.md`。三者で食い違う場合は **02_フェーズとガバナンス → 本ファイル → CLAUDE.md 要約** の順で正とする。

## データとプライバシー

- 個人・顧客を特定できる情報（実名・メール全文・契約ID・非公開URLなど）はチャットや成果物に**そのまま載せない**。例示はダミーかマスク。
- APIキー・トークン・パスワード・口座の全桁は**扱わない**（設定はユーザー環境の秘密管理に任せる）。

## 実行の境界

- **削除・ディスク整理の実行・一括メール送信・Slack/Notion の本番投稿・課金APIの実行**は、エージェントが手順・文案・コマンド案に留めることがある。**実行はユーザーが確認してから**。
- CRM・会計・人事データの更新は、**影響件数・競合ルール**を先に示し、合意済みの操作に限定する。

## 品質とリスク

- Web 取得・調査では**出典URLと取得日**を残す（可能な範囲）。
- 数値・契約・法務・労務・税務に効く内容は**仮・要確認**とし、正本・専門家確認を促す。
- 推測で「決定事項」を作らない。**要確認**で切る。

## スキル追加・更新時（3点チェック）

1. **README 早見表**に1行追加（ファイル名・コマンド風・一言・典型シーン）。
2. フロントマターに `id`（ファイル名と同一・kebab-case）、`domains`、`requires`、`related` を入れる。
3. **前提条件**・**使わないほうがよい場合**・**完了の目安**を必ず書く（環境の取り違え・手戻り防止）。複数成果物がある場合は **`post-meeting.md` と同様の表**（成果物・形式・保存先の例）を載せてよい。
