# /post-meeting — 会議あとのフォロー（議事録〜文案）

## これは何か

会議が終わったあとにやりがちな**議事録のたたき台**に加え、御礼メール案、Slack 用の短文、Notion のプロパティ案などを**まとめて下書き**します。実際の送信・投稿は、ユーザーが確認してからにします。

## いつ使うか

- 議事録と「送る文」を**セットで**欲しいとき
- カレンダーと Notion、Slack があって、**会議後の型**を決めたいとき

## 使わないほうがよい場合

- **議事録本文だけ**でよい → `auto-minutes` か `universal-meeting-minutes`
- 連携ツールがなく、コピペ用の文面だけなら、軽めの依頼なら `universal-meeting-minutes` でも足りることがある

## 前提条件

- **MCP や gogcli** が使えると、案だけでなく反映手順まで出しやすい。使えない場合は**コピペ用ブロック**で完結させる。

## 想定する成果物（例）

複数のアウトプットを**表で押さえる**と、依頼と保管ルールがずれにくいです。  
下表のパス・チャンネル・DB 名は**運用の例**。`clients/` や顧客名は**自分のフォルダ規約に読み替え**、機微は匿名化する。実際の保存先・チャンネルは**ユーザーと確定**すること。

| # | 成果物 | 形式 | 保存先（例） |
|---|--------|------|----------------|
| 1 | 議事録（プレーンテキスト） | `.md` | `clients/顧客名/minutes/` など |
| 2 | 議事録（フォーマット済み） | `.docx` | 同上（または共有ドライブの議事録フォルダ） |
| 3 | 御礼メール下書き | Gmail 下書き | Gmail の下書き |
| 4 | Slack 社内報告 | Slack メッセージ | `#meeting-notes` など |
| 5 | Notion DB 更新 | DB レコード | Notion の Meetings DB など |

MCP や API が使えない場合は、3〜5 は**チャット上の文案**として渡し、ユーザーが手で下書き・投稿する。

## どう依頼するか

1. **`post-meeting`** と、文字起こし・メモ・ファイルを渡す。
2. 日付や**商談か社内か**など、文脈を短く。
3. 欲しい成果物を列挙（例: 議事録 md・御礼メール・Slack 3行・Notion の項目案）。

**例:** 「`post-meeting`。トランスクリプト貼る。議事録 md、御礼メール、Slack 3行。Notion は商談 DB のプロパティ案だけ。」

## 進め方（エージェント向け）

1. 入力と会議のメタ情報をそろえる。足りなければ質問。
2. 議事録（`.md` など）を書く。必要なら Word 化の手順も添える。
3. 依頼どおりメール・Slack・Notion 案を**別ブロック**で出す。
4. MCP で反映できるなら手順まで。無理ならコピペ用にとどめる。
5. 秘密や固有情報はマスクする。

## 完了の目安

- 依頼に含まれる成果物が、**表のどれに当たるか**がひと目で分かる（不要な行は出力しない）
- 議事録に**決定事項と ToDo** が含まれる
- Slack・メールごとに**長さとトーン**が合っている
- **勝手に送信・投稿していない**（案と手順まで）
