# Obsidianインプット：RSS・LINE の動線（設計メモ）

Obsidian をハブにしたときの **RSS** と **LINE** の取り込みについて、意図・設計の型・注意点を1か所にまとめる。実装は段階的でよい。

---

## まず：Skills の symlink は「不要」か

**必須ではない。** 動線の本質は「スキル Markdown が、編集時と実行時で同じ内容か」だけ。

- **正本が1つ**（例: Vault の `.claude/skills/` のみ、または `~/.claude/skills/` のみ）で、使うツールが常にそこを読むなら、symlink はなくてよい。
- **二系統**（Vault にコピー＋ホームに別実体）でメンテが続くなら、図のように symlink で一本化するのは**有効な整理**。

結論：**「なくて困る」ものではなく、二重管理が痛いときの処方箋。**

---

## 共通の設計意図（RSS も LINE も同じ）

| 意図 | 説明 |
| --- | --- |
| **入口の分離** | タイムライン全取り込みではなく、「自分が選んだ入口」だけを Vault に流す。 |
| **生ログの長期放置を防ぐ** | 取り込み先は原則 `20_Input/` 系。資産化は別ステップ（要約・タグ・ドメイン振り分け）。正本の考え方は [[04_AI/01_ツール運用/05_データ蓄積ガイド]]。 |
| **失敗しても回る** | 取得失敗時はスキップ・手動貼付にフォールバック。週1で詰まりを見る。 |
| **機微の分離** | 個人メモと業務（会員・取引先）を同じ自動パイプに混ぜない。業務は別ガバナンス（[[02_Business/RINGBELL/00_索引]] 等）。 |

---

## RSS 動線

### 何のためか

- **アルゴリズムフィードではなく購読リスト**で情報を取る。
- 「あとで読む」ではなく **Vault に置いた時点で検索・リンク・再編集の対象**にする。
- 定点観測（特定メディア・ブログ・YouTube チャンネル等）と相性が良い。

### 設計の型（候補）

実装は次のいずれか、または併用。

1. **Obsidian プラグイン**  
   Local RSS 系でフィードからノート生成。設定は Obsidian 内に閉じ、起動時・手動更新。

2. **外部クローラ＋フォルダ投入**  
   小さなスクリプト（Python / n8n 等）で RSS を fetch → Markdown を `20_Input/` 以下に書く。Cron で日次。

3. **「URL だけ」先に溜める**  
   RSS 全文取得はせず、Readwise / Clipper / 手動でリンク＋1行メモだけ入れ、週次で処理（運用コスト最小）。

### 格納先の目安

| 内容 | 置き場の例 |
| --- | --- |
| 記事メタ＋抜粋＋自分メモ | `20_Input/02_Clippings/` または `20_Input/03_RSS/`（新設してもよい） |
| YouTube 新着要約パイプライン | バックログ案と同じく [[04_AI/01_ツール運用/06_自動化バックログ]]（YouTube チャンネルRSSの節）と整合 |

### 前提・リスク

- **利用規約・robots・取得間隔**：スクレイピングで全文を抜く場合は特に注意。RSS が本文を載せていない場合はリンク＋要約に留めるのが安全。
- **ノイズ**：フィード数を増やしすぎるとインボックス地獄。購読は **意図的に少なく**、週次レビューで削る。
- 個人の入口戦略の候補リストは [[05_Life/15_個人自動化候補一覧]] と揃えるとよい。

### 最小の最初の一歩

- 購読リストを **Markdown で1ファイル**（ドメイン別URL＋なぜ読むか1行）に固定する。  
- 次に、**1フィードだけ** 自動取り込み（または週1手動で「未読分」をノート化）を試す。

## LINE 動線

### 何のためか

- LINE 上のメモ・転送・自分宛けの断片を、**Obsidian 上で検索・リンク・バージョン管理**できる形に寄せる。
- 「LINE の検索で探す」から「Vault の検索で探す」へ移すのが主目的（業務の公式 LINE 運用とは別レイヤー）。

### 業務 LINE と切り分け

| 種類 | Vault 自動投入 |
| --- | --- |
| **個人**（Keep、自分用メモ、自分に送ったリンク等） | 設計対象。機微はマスク or 別フォルダ。 |
| **副業・本業の公式 LINE（会員対応）** | 自動で個人 Vault に流さないのが原則。業務正本・ガバナンスは各ドメインの設計に従う。 |

### 設計の型（現実的な候補）

公式が「Obsidian 直結」を提供しているわけではないので、次のどれかで組む。

1. **手動・準手動の逃がし口（最も堅い）**  
   週1、Keep やトークから **コピペ or 共有 → `20_Input/`** に集約。テンプレで日付・出所（LINE）を1行固定。

2. **メール橋渡し**  
   LINE の通知や Keep のメール系機能が使えるなら、**専用メール → メール取得 → Markdown**（実装コストとセキュリティを要検討）。

3. **サードパーティ・自動化**  
   IFTTT / Make / 自前 Webhook 等。**トークン・会話ログの扱い**と利用規約を必ず確認。個人用途に限定。

4. **「LINE に置かない」運用への寄せ**  
   リンクと一行メモは最初から Obsidian に書く。LINE は通知専用。構築コストはゼロに近い。

### 格納先の目安

| 内容 | 置き場の例 |
| --- | --- |
| 個人メモの取り込み | `20_Input/04_LINE/`（新設）または `00_Inbox/` 経由で仕分け |
| 資産化後 | `05_Life/` 等、ドメインに応じて移動（[[04_AI/01_ツール運用/05_データ蓄積ガイド]]） |

### 前提・リスク

- **会話ログの自動流出**は最も危険。フィルタ（自分のメモだけ、グループ除外等）を最初に決める。
- **同期の二重化**（Google Drive の Vault と別経路で同じファイルを触る）に注意。競合時の正本は Git か Drive のどちらかに統一。

### 最小の最初の一歩

- **「LINE から Obsidian に逃がす」週1ルーティン**（15分）と、**置き場1フォルダ**だけ決める。  
- 自動化は、そのルーティンで「毎回同じ形式」が固まってから載せる。
