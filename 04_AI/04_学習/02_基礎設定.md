# Claude Code 完全セットアップガイド
**非エンジニア向け・完全版**

> **このガイドについて**  
> 複数の実践者・開発者・Anthropic 公式情報をもとに構成した、非エンジニアが Claude Code を最大限に活用するための完全リファレンス。初回セットアップから日常運用・トークン管理・自動化まで、一冊で完結する。

---

## 結論から言う — Claude Code は「必須」か？

**非エンジニアにとって Claude Code は必須ではないが、使いこなせると仕事の生産性が根本から変わる。**

ChatGPT や通常の Claude との決定的な違いはこの一点だ。

| | 通常の Claude / ChatGPT | Claude Code |
|---|---|---|
| できること | テキストを返す | ファイルを直接編集・コマンドを実行・外部サービスと連携 |
| 記憶 | 毎回リセット | 設定ファイルで「記憶」を持たせられる |
| 価値の源泉 | 入れた瞬間が最大値 | 育てるほど価値が上がる |
| リスク | ほぼゼロ | 設定を誤ると実害が出る（ファイル削除・情報漏洩） |

**なぜ非エンジニアにこそ重要か。**  
エンジニアはコードで自動化できる。非エンジニアはこれまでできなかった。Claude Code は「プログラムを書けなくても、AI に実際に作業させる」唯一の現実的な手段だ。記事執筆・資料作成・メール処理・データ整理——これらを「指示するだけで動かす」状態を作れる。

**導入の重要性。**  
ただし、入れるだけでは意味がない。Claude Code の本当の価値は「初期設定と育成」にある。設定を何もしなければ ChatGPT と大差ない。このガイドに書いてあることを実装することで初めて、「毎回説明し直す道具」から「文脈ごと理解している自律的な部下」に変わる。

<a name="原則"></a>
## 設定前に理解すべき2つの原則

### 原則① Claude Code はステートレスだ

起動するたびに「初めまして」の状態から始まる。昨日の会話も、先週の指示も、すべてリセットされる。毎回「このプロジェクトはこういう構成で、日本語で答えて、結論から述べて…」と説明し直しているなら、設定ができていない証拠だ。

この問題を解決するのが `.claude` フォルダと `CLAUDE.md` だ。ここに書いたことは Claude が毎回自動で読み込む。「初日から文脈を知っている即戦力」として起動する状態を最初に作る。

### 原則② コンテキストウィンドウは有限リソースだ

Claude Code には「作業机の広さ」にあたる制限がある（最大100万トークン）。会話を続けるほど、過去のやり取りが机を占領し、実際の作業に使えるスペースが減る。

```
トークン消費の実態:
 5件の会話:  7,500 トークン
10件の会話: 27,500 トークン
20件の会話: 105,000 トークン（1件目の31倍）

実測では総消費の 98.5% が「過去の読み返し」に使われ、
実際の出力に使われたのはわずか 1.5%
```

「最近 Claude がバカになった」と感じたら、原因はモデルの劣化ではない。コンテキストが溢れて「古い情報に溺れている」状態だ。設定ファイルは短く保ち、定期的にリセットすることが品質維持の鍵になる。

<a name="step1"></a>
## Step 1 — 使用環境の準備

### 非エンジニアには Cursor を推奨する

| 環境 | 対象 | 特徴 |
|------|------|------|
| **Cursor（推奨）** | 非エンジニア | ファイルプレビュー・複数ターミナル分割・エラーをその場で質問できる |
| ターミナル単体 | エンジニア | 軽量・SSH 環境でも動く |
| VS Code 拡張 | VS Code ユーザー | 既存環境に組み込める |

理由はシンプルだ。Claude Code はターミナル（黒い画面）で動く。非エンジニアにとってターミナル単体は見通しが悪い。Cursor を使うと、ターミナル・ファイルプレビュー・AI エージェントを1画面に収められる。分からなかったことをターミナルからそのまま左側のチャットに貼って質問できる利点も大きい。

**推奨レイアウト**

```
┌──────────────────┬────────────────┐
│ 左: Cursor Agent │ 右上: ファイル  │
│（質問・相談用）   │ プレビュー      │
│                  ├────────────────┤
│                  │ 右下: ターミナル │
│                  │（Claude Code）  │
└──────────────────┴────────────────┘
```

<a name="step2"></a>
## Step 2 — .claude フォルダを設計する（核心）

これがすべての起点だ。ここを正しく設計するかどうかで、Claude Code が「命令待ちの道具」になるか「自律的に動く部下」になるかが決まる。

### .claude フォルダの全体像

```
your-project/
├── CLAUDE.md              # セッション開始時に最初に読まれる「脳の設定書」
├── CLAUDE.local.md        # 自分だけの個人設定（.gitignore に追加）
└── .claude/
    ├── settings.json      # 権限設定（git 管理可）
    ├── commands/          # カスタムスラッシュコマンド（手動呼び出し）
    ├── rules/             # CLAUDE.md が肥大化したときの分割先
    ├── skills/            # Claude が自律的に判断して起動するワークフロー
    └── agents/            # 専門分業の「部下エージェント」
```

---

### ① CLAUDE.md — セッション開始時に必ず読まれる指示書

Claude Code はセッション開始時、真っ先に CLAUDE.md を読み込む。以降ずっとその内容を頭に入れたまま動く。ここに書いたことは Claude が必ず守る。

**200行以内に収めること。** それ以上になると指示の遵守率が実際に下がる。長くなったら `rules/` フォルダに分割する。

**書く内容 / 書かない内容**

| ✅ 書く | ❌ 書かない |
|---------|-----------|
| 出力言語・口調・フォーマット | ReadMe と重複する内容 |
| 禁止事項（公開してはいけない情報など） | 長い理論的説明 |
| よく使うコマンドやツールの優先順位 | リンクで代替できるドキュメント |
| 完了時の振る舞い（通知音を鳴らす、等） | AI に自動生成させた冗長な文章 |

**削除しても Claude の動作が変わらない行は、その行が不要だということだ。**

**CLAUDE.md に完了条件・検証手段を書くと精度が上がる**（Anthropic 公式ドキュメントでも推奨）:

```markdown
## 完了条件
- コードを変更したら必ず npm test を実行する
- 型チェックは npm run typecheck で確認する
- 変更前に git status で対象ファイルを確認する
```

完了条件を明示することで Claude が「何をもって完了とするか」を自律的に判断できるようになる。

**CLAUDE.md テンプレート（非エンジニア向け）**

```markdown
# 基本設定

## 回答スタイル
- 必ず日本語で回答する
- 結論を先に述べ、前置き・挨拶・同調は不要
- 絵文字・装飾は使わない
- 完了したら通知音を鳴らす
- 間違いや改善点は遠慮なく指摘する

## 禁止事項
- .env ファイルの中身をチャットに表示しない
- 確認なしにファイルを削除しない
- git push は必ず確認を求めてから実行する

## ツール優先順位
- YouTube URL が来たら Gemini YouTube スキルを使う
- X（Twitter）の URL が来たら Grok サーチを使う

## 作業フロー
- 複雑なタスクは実行前にプランファイルを作成し、確認を求める
- プランファイルには意思決定の根拠を含める
```

**CLAUDE.md は階層ごとに設計する**

1ファイルに集約せず、フォルダ階層ごとに配置する。

```
~/work/
├── CLAUDE.md          ← 仕事全体の共通ルール
├── clients/
│   ├── CLAUDE.md      ← クライアント対応のルール
│   └── client-A/
└── content/
    └── CLAUDE.md      ← コンテンツ制作のルール
```

### ② .claude/commands/ — カスタムスラッシュコマンドを作る

`commands/` フォルダに Markdown ファイルを1つ置くだけでカスタムコマンドが生まれる。毎回手で打っていた長い指示が1コマンドに集約される。

| ファイル名 | 使えるコマンド |
|-----------|-------------|
| `review.md` | `/project:review` |
| `summarize.md` | `/project:summarize` |
| `post-writer.md` | `/project:post-writer` |

ファイル内で `` !`コマンド` `` の記法を使うと、シェルコマンドの実行結果をそのまま Claude のプロンプトに埋め込める。

**コードレビューコマンドのテンプレート**（`.claude/commands/review.md` として保存）

```markdown
---
description: マージ前にカレントブランチの差分をレビューする
---
## 変更対象ファイル

!`git diff --name-only main...HEAD`

## 詳細な Diff

!`git diff main...HEAD`

以下の観点でレビューしてください：
1. コード品質の問題
2. セキュリティの脆弱性
3. テストカバレッジの欠落
4. パフォーマンスへの懸念

ファイルごとに具体的なフィードバックを出してください。
```

### ③ .claude/skills/ — Claude が自律的に起動するワークフロー

commands との違いが重要。

| | commands | skills |
|---|---|---|
| **起動方法** | 自分がスラッシュコマンドで呼び出す | Claude が会話を読んで自律的に判断して動かす |
| **呼び出し** | 手動 | 自動 |

「PR のセキュリティチェックして」と言うだけで、Claude が「これは security-review スキルを使う場面だ」と判断して自動でワークフローを走らせる。呼び出しすらいらない。

スキルはフォルダ単位（パッケージ）で管理できる点も大きい。

```
.claude/skills/
├── security-review/
│   ├── SKILL.md           # スキルの定義
│   └── DETAILED_GUIDE.md  # 詳細な基準
└── post-writer/
    ├── SKILL.md
    └── style-reference.md # 文体のリファレンス
```

**スキルの作り方（最も簡単な方法）**

```
Claude Code に: 「この作業が気に入ったのでスキル化してください」
→ 自動でスキルファイルが生成される
```

**スキルの育て方**: 使うたびに気づいたルールを追記する。「この表現は使わないで」「このフォーマットで」という指摘をファイルに反映するたびに精度が上がる。

**スキルの粒度**: 記事執筆とサムネイル作成は別スキルにする。「不要な工程を切り離せる粒度」が目安。

**セキュリティレビュースキルのテンプレート**（`.claude/skills/security-review/SKILL.md`）

```markdown
---
name: security-review
description: セキュリティ監査。コードレビュー時、デプロイ前、
  またはユーザーがセキュリティに言及した際に使用する。
allowed-tools: Read, Grep, Glob
---
以下の観点でセキュリティ脆弱性を分析してください：

1. SQLインジェクションと XSS のリスク
2. 認証情報やシークレットの露出
3. 安全でない設定
4. 認証・認可のギャップ

深刻度の評価と具体的な修正手順を含めてレポートしてください。
```

---

### ④ .claude/agents/ — 専門分業の「部下エージェント」

複雑なタスクを専門家に分業させる。各エージェントは**独立したコンテキストウィンドウ**で動くため、メインセッションが汚染されない。トークン節約にも直結する。

```
.claude/agents/
├── code-reviewer.md      # コードレビュー専門
└── security-auditor.md   # セキュリティ監査専門（Read 権限のみ）
```

- **ツール権限を個別に設定できる**。セキュリティ監査エージェントには Read・Grep・Glob だけ持たせ、書き込みは絶対させない設計が安全
- **`model` フィールドでモデルを指定できる**。単純な読み取り探索は Haiku で十分。Sonnet・Opus は本当に必要な場面だけに使う

---

### ⑤ .claude/rules/ — CLAUDE.md 肥大化対策

CLAUDE.md が 200 行を超えてきたら、`rules/` フォルダに役割ごとに分割する。

```
.claude/rules/
├── writing-style.md   # 文章スタイル規約
├── tools.md           # ツール優先順位
└── security.md        # セキュリティルール
```

担当者ごとに管理でき、互いに干渉しない。

---

### CLAUDE.md・スキル・エージェントの関係

```
CLAUDE.md = 会社のミッション・パーパス（全タスクに自動適用）
    └── skills/ = 各部署の業務マニュアル（呼び出した時だけ適用）
        └── agents/ = 専門分業の部下（独立したコンテキストで並列稼働）
    └── commands/ = 定型業務の手順書（手動で呼び出す）
    └── rules/ = CLAUDE.md の補足資料（肥大化した時の分割先）
```

<a name="step3"></a>
## Step 3 — フォルダ構成を整える

Claude Code の精度はフォルダ構成と直結する。デスクトップに散乱したファイルでは性能が発揮されない。

### 推奨フォルダ構成

```
~/work/
├── CLAUDE.md               ← 仕事全体の共通ルール
├── clients/
│   ├── CLAUDE.md
│   ├── client-A/
│   └── client-B/
├── dev/
│   ├── CLAUDE.md
│   └── projects/
├── personal/
├── content/
│   ├── CLAUDE.md
│   ├── articles/
│   └── media/
└── agents/                 ← AI エージェント設定・アーカイブ
```

### 設計のルール

- 作業の種類ごとにフォルダを分け、それぞれに CLAUDE.md を配置する（4〜5 階層でも問題ない）
- **デスクトップや親フォルダに作業フォルダを置かない** — 顧客リスト・確定申告データ等の無関係なファイルまで読み込まれるリスクがある
- Cursor でプロジェクトを開くときは、該当フォルダを `Open Project` で指定する

<a name="step4"></a>
## Step 4 — スキルを作る

Step 2 で `.claude/skills/` の設計思想は説明した。ここでは運用の実践に絞る。

### 最初に作るべき3つのスキル

どんな用途で使う人にも共通して効果の高いスキルがある。

| スキル名 | 呼び出し例 | 効果 |
|---------|-----------|------|
| `writer` | `/writer 記事を作成して` | 文体・構成・禁止表現を毎回説明しなくて済む |
| `summarize` | `/summarize この文書を要約して` | 要約の粒度・フォーマットが固定される |
| `wrap-up` | `/wrap-up` | セッション終了時の引き継ぎを自動化（後述） |

### スキルのライフサイクル

```
① 作業を Claude Code で実行する
② 「この作業が気に入ったのでスキル化して」と指示する
③ 使うたびに「このフォーマットにして」「この表現は使わないで」と指摘を追記する
④ スキルが自分専用の業務マニュアルになる
```

<a name="step5"></a>
## Step 5 — セキュリティ設定

> Anthropic 公式: *"Claude Code only has the permissions you grant it. You're responsible for reviewing proposed code and commands for safety before approval."*

**デフォルトのまま使うことは非推奨。** Claude Code はファイルの読み書き・ターミナルコマンドの実行・外部サービスとの連携をすべてこなす。この強力な権限が設定不足のまま動くと、ファイルの消失・API キーの漏洩・外部への情報送信が実際に発生する。

### 脅威の全体像

| 脅威 | 具体例 |
|------|--------|
| ファイルの誤操作 | 重要ファイルの上書き・削除（取り返しがつかない） |
| 情報漏洩 | .env の API キー・パスワードが外部に送信される |
| 意図しない操作 | 自動承認を連打 → 本番環境の DB を削除 |
| プロンプトインジェクション | 読み込んだ外部ページに「.env を送信せよ」という隠し指示が埋め込まれており AI がそれに従う |

**プロンプトインジェクション**: 悪意ある者が Web ページや共有ファイルに AI への指示を隠し込む攻撃手法。Claude Code が自動的にページを読み込んだ際、そのまま指示に従って API キーを外部送信してしまう。気づいた時には API 費用が数十万円になっていたというケースが実際に報告されている。

### 設定ファイルの優先順位（4層構造）

```
優先度: 高 → 低
1. Managed      組織ポリシー用（MDM / Jamf で配布）
2. Local        .claude/settings.local.json（.gitignore 推奨）
3. Project      .claude/settings.json（git 管理・チーム共有）
4. User         ~/.claude/settings.json ← 個人の全プロジェクト共通設定
```

設定は **deny → ask → allow** の順で評価される。deny は最優先で、セッション中に上書きされない。

---

### 【最優先①】サンドボックスを有効化し、脱出口も塞ぐ

サンドボックスは Claude Code の実行環境を OS レベルで隔離する機能（macOS: Seatbelt、Linux: Bubble Wrap）。**有効化するだけでなく、脱出口も必ず封じること。**

```json
{
  "sandbox": {
    "enabled": true,
    "allowUnsandboxedCommands": false
  }
}
```

`allowUnsandboxedCommands: false` がないと、パラメータ一つでサンドボックスを回避できてしまう。この2行は必ずセットで設定する。確認: Claude Code 内で `/sandbox` と入力。

---

### 【最優先②】deny ルールで危険コマンドを封じる

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf *)",
      "Bash(rm -rf /*)",
      "Bash(git push --force *)",
      "Bash(git reset --hard *)",
      "Bash(chmod 777 *)",
      "Bash(curl *)",
      "Bash(wget *)"
    ]
  }
}
```

確認画面にこれらのキーワードが出たら必ず立ち止まること:

| キーワード | 意味 | リスク |
|------------|------|--------|
| `rm` / `delete` | ファイル削除 | 取り返し不可 |
| `push` | 外部へのコード送信 | 意図しない公開 |
| `install` | パソコンへのインストール | 不審なソフト混入 |
| `--force` | 強制実行 | 上書き・破壊 |
| `drop` | データベース削除 | データ消滅 |

---

### 【最優先③】機密ファイルの読み取りをブロック

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(**/*.pem)",
      "Read(**/*.key)"
    ]
  },
  "sandbox": {
    "filesystem": {
      "denyRead": ["~/.aws/credentials", "~/.ssh"]
    }
  }
}
```

`permissions.deny` は Claude の Read ツール経由をブロック。`sandbox.filesystem.denyRead` は Bash コマンド（`cat ~/.ssh/id_rsa` 等）経由もブロック。**両方設定して初めて二重の壁になる。**

---

### 【重要】.env ファイルの適切な管理

1. **チャット欄に API キーを直接貼らない** — Anthropic のサーバーに送信され、漏洩リスクがある
2. **API キーは .env に書き、Claude Code には「.env から読んで」と指示する** — Claude Code 自身が .env を読まなくても、コード実行時は環境変数として利用可能
3. **.env を .gitignore に追加する** — Git にコミットすると公開リポジトリで世界中から見えてしまう

```bash
# .gitignore に追記
.env
.env.*
```

**漏洩した場合の対応**: 即座に該当サービスの管理画面でキーを無効化し、新しいキーを再発行する（クレジットカードの紛失時と同じ対応）。

---

### 【重要】ファイル履歴バックアップのリスク

Claude Code はファイル編集時に自動バックアップを `~/.claude/file-history/` に保存する。暗号化されていない。うっかり .env を編集させると、後から deny 設定をしてもバックアップには残り続ける。

対策: 定期的に中身を削除する（「file-history フォルダを削除して」と Claude Code に依頼するだけで実行できる）。

---

### 【推奨】ネットワークアクセスをホワイトリスト制限する

プロンプトインジェクション対策として最も効果的。許可ドメイン以外への通信を遮断することで、悪意ある指示による情報送信を防ぐ。

```json
{
  "sandbox": {
    "network": {
      "allowedDomains": [
        "github.com",
        "npmjs.com",
        "pypi.org",
        "registry.npmjs.org"
      ]
    }
  }
}
```

---

### 【推奨】hooks で外部送信コマンドを監視する

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{ "type": "command", "command": "~/.claude/hooks/validate-command.sh" }]
      }
    ]
  }
}
```

- `exit 2` でコマンドをブロック
- `exit 0` で警告のみ通過（作業を止めずに確認させる）

---

### 【推奨】bypassPermissions の使い方

`bypassPermissions: true` にすると確認ダイアログが消えて作業速度が上がる。**必ず deny ルールを設定してから** 有効にすること。deny に入っているコマンドは bypassPermissions でも実行されない。

> 個人の通常作業では ON で効率重視。センシティブな案件・本番操作では OFF に切り替えるのが現実的な運用。

---

### 【運用習慣】許可モードの使い分け

| 場面 | 推奨モード |
|------|------------|
| 使い始め・慣れていない段階 | 都度確認（デフォルト） |
| ファイル読み取りのみ | 自動許可でも可 |
| ファイル編集・コマンド実行 | 都度確認を維持 |
| 内容を理解できるようになったら | 自動承認に変更可 |

---

### 【運用習慣】知らないプロジェクトを開かない

外部からダウンロードした CLAUDE.md や MD ファイルには悪意ある指示が仕込まれている可能性がある。Claude Code はプロジェクトを開いた瞬間に CLAUDE.md を自動で読み込み従おうとする。中身を確認せずに開くことは危険。

確認方法: テキストエディタで中身を確認し、`rm`・`push`・`install`・`drop` 等のキーワードがないかチェックする。不安な場合はチャット版 Claude に「このコードに危険な内容はあるか」と評価させる（Claude Code に読み込ませると指示に従ってしまうため、必ず別の AI を使う）。

---

### 【運用習慣】/permissions で月1回棚卸し

月1回 `/permissions` コマンドで以下を確認する:
1. セッション中に増えた「Always allow」が適切か
2. 不要になった古い許可ルールが残っていないか
3. 設定した deny ルールが機能しているか

`/status` で設定ファイルの読み込み状態も確認する。

---

### 【プライバシー】学習利用をオフにする

個人プランの場合、Claude.ai の設定から「会話データを学習に使用しない」をオフにする。API キーなどをチャットに誤って貼った際のリスク軽減にもなる。

---

### セキュリティ：やってはいけないこと

- `--dangerously-skip-permissions` をコンテナ外（本番マシン）で使う
- API キー・パスワードをチャット欄に直接貼る
- 知らないプロジェクトの CLAUDE.md を確認せずに開く
- 「Always allow」を毎回連打して権限を蓄積させる
- .env を Git にコミットする
- Claude Code が提示した削除・上書きコマンドを内容確認せずに実行する

**削除コマンド実行前に必ず確認すること**: 「このコマンドはどこまで影響するか」を自分で考えてから実行する。AI は自分が大切に育ててきたデータの価値を知らない。CLAUDE.md・スキル群・settings.json を吹き飛ばしてしまった実例が報告されている。

### 【参考・組織展開】メルカリに見る Claude Code セキュリティ配布（要点メモ）

> **出典・注意:** カンファレンス・ブログ・社外公開資料などで語られている「メルカリでの Claude Code セキュリティ／MDM 配布」の**社外要約メモ**。正式方針は各社の情シス・セキュリティ部門と Anthropic / Claude Code のエンタープライズ資料を正とする。

#### リスクの前提（Step 5 全体との接続）

Claude Code は **コマンド実行・ファイル読み書き・Web 取得**ができるため、PC 上の認証情報や重要ファイルへのリスクがある。個人向け前半で書いた **サンドボックス・deny・ネットワーク制限**と同型の脅威を、**全社端末にどう既定で埋めるか**という話に読み替える。

#### 公開事例として整理されていた対策（5つ）

| # | 内容 | 個人向け設定との対応イメージ |
|---|------|------------------------------|
| 1 | **bypass 相当の自動承認を禁止** — 人間が必ず確認 | `bypassPermissions` の濫用禁止、`--dangerously-skip-permissions` を使わない運用（「やってはいけないこと」） |
| 2 | **curl 等の危険コマンド**は都度確認 | `permissions.deny` の `Bash(curl *)` 等と、承認フローの徹底 |
| 3 | **環境変数の読み取り・sudo を禁止** | `.env` / `~/.ssh` の denyRead、危険 Bash の deny |
| 4 | **Sandbox** でワークスペース外アクセス・ネットワークを制限 | 「サンドボックスを有効化し脱出口を塞ぐ」「ネットワークのホワイトリスト」節 |
| 5 | **組織のシステムプロンプト（司令塔文書）にセキュリティポリシーを埋め込む** | 企業版 CLAUDE.md／ポリシーへのリンク・禁止事項・インシデント連絡先 |

#### 全社員への届け方：MDM 等での一斉配布

- **MDM（端末管理）**で Managed 設定を配ると、社員が手を抜かなくても**ベースラインが安全側**に寄る。
- 本書の **設定ファイルの優先順位**（`Managed > Local > Project > User`）どおり、**Managed は最優先**で、**社員が端末上で恒久的に緩められない**ことがある（製品・MDM の組み合わせによる）。

#### 張力：エンジニア vs 非エンジニア

| 側 | 求めがちなこと |
|----|----------------|
| エンジニア | **ガードレールは維持**しつつ、プロジェクトごとのカスタマイズや検証用の緩みを残したい |
| 非エンジニア | **最初から最も安全な既定**で迷わず使いたい |

**1つの設定ですべてを満たすのは難しい**、という整理が紹介されていた。

#### 事例としての解：属性別の二層（多層）配布

- MDM から **社員属性（エンジニア / 非エンジニア等）を判別**し、**別の設定セット**を流す。
- **エンジニア向け:** 安全性を保ちつつ、業務上必要な範囲でカスタム可能なプロファイル。
- **非エンジニア向け:** 最も厳しい既定のプロファイルをそのまま提供。

#### 使いどころ

大企業で Claude Code を **全社導入**するときの「**設定を誰にどう配るか**」議論の材料。**一律ロールアウトだけでは折り合いがつかない**場合に、**属性別プロファイル + MDM 一斉配布**を設計案として持ち出せる。

## Step 6 — パフォーマンス設定

### 思考予算（effortLevel）を上げる

デフォルトでは Pro / Max プランの effortLevel は `medium`（API・Team・Enterprise は `high` のまま）。これが「最近 Claude がバカになった」と感じる原因の一つ。

`~/.claude/settings.json` に追記:

```json
{
  "extendedThinking": true,
  "effortLevel": "high"
}
```

環境変数でも設定できる:

```bash
export MAX_THINKING_TOKENS=31999
```

コストは上がるが、修正の手戻りが減るため結果的にコスト削減になる。

---

### ステータスラインを表示する

CCSL 等のツールを導入すると、ターミナル下部にコンテキスト使用率・残り時間がリアルタイムで表示される。

```
[compact] session: 5h 32m | ctx: 87% | ████████░ ]
```

80% で黄色信号、90% 超で `/clear` を検討するタイミング。

---

### 環境変数：追加で設定する4つ

```bash
# ① VS Code 拡張とターミナルでログが分散するのを防ぐ
export CLAUDE_CONFIG_DIR="$HOME/claude-data"

# ② Claude がターミナルタブ名を上書きするのを無効化（複数セッション管理時）
export CLAUDE_CODE_DISABLE_TERMINAL_TITLE=1

# ③ 複数インスタンスを並列協調させる実験的機能を有効化
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# ④ サブエージェントは軽量モデルで実行してコスト削減
export CLAUDE_CODE_SUBAGENT_MODEL=haiku
```

---

### その他の設定

```json
{
  "ENABLE_TOOL_SEARCH": "auto:5",
  "cleanupPeriodDays": 365
}
```

- `ENABLE_TOOL_SEARCH: "auto:5"` — MCP ツールが多い場合、起動時のコンテキスト消費を抑える
- `cleanupPeriodDays: 365` — セッションログを30日ではなく1年保持。過去の作業を grep で検索できるようになる

**settings.json に `$schema` を追加すると VS Code で補完が効く:**

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json"
}
```

<a name="operations"></a>
## コンテキスト管理 — 「バカになった」を防ぐ

Claude Code を使い続ける上で最も重要な習慣。これを怠ると「Claude がバカになった」状態が常態化する。

### 主要コマンド一覧

| コマンド | 用途 | 使うタイミング |
|----------|------|-------------|
| `/clear` | コンテキストを完全リセット | タスク切り替え時・2時間超えた時 |
| `/compact [指示]` | 会話を圧縮（重要情報を選んで残す） | 文脈を維持したまま節約したい時 |
| `Esc × 2` | 直前の状態に巻き戻し | Claude が変な方向に進み始めた時 |
| `/rewind` | 過去の特定メッセージに復元 | 細かく巻き戻したい時 |
| `/context` | 現在の使用率を確認 | 状態確認 |
| `/permissions` | 許可ルールの一覧表示 | 月1回の棚卸し |
| `/sandbox` | サンドボックス状態の確認 | 初期設定確認 |
| `/status` | 設定ファイルの読み込み確認 | トラブルシューティング |

### /clear と /compact の使い分け

```
/clear を使う場面:
  - タスクが完全に切り替わった
  - 2 時間以上同じセッションを継続した
  - 同じミスを 2 回繰り返した（3 回目の前に必ずクリア）
  - コンテキストが 90% を超えた
  ※ /clear 前に「今の進捗をファイルに書き出して」と指示してからリセット

/compact を使う場面:
  - 文脈は残したいが軽くしたい
  - 「この修正はあと少し」という状態
  ↓ 指示付きで使うのが推奨
  /compact エラーの原因と解決策だけ残して
  /compact 今のタスクに関係ある部分だけ
```

### 日常の運用ルール

1. **タスク切り替え時は必ず `/clear`** — 「ついでにこっちも」が品質低下の元凶
2. **調査はサブエージェントに委譲** — メインのコンテキストは指示と実装だけに使う
3. **2 回修正して直らなかったら `/clear`** — 3 回目はコンテキスト汚染が原因
4. **`/compact` は必ず指示付きで使う** — 引数なしだと重要な情報まで消える
5. **CLAUDE.md に全ルールを書く** — `/clear` してもルールが自動復元されるように
6. **2 時間超えたら一度リセット** — 長時間セッションは確実に品質が劣化する

### メモリを積極的に育てる

オートメモリはデフォルトで有効。「覚えておいて」と明示的に伝えることで精度が上がる。

```
覚えさせるべき内容:
- 自分の作業スタイルの好み
- フィードバックの傾向（「この表現は使わないで」）
- プロジェクトの固有名詞・人物情報
- 判断の条件分岐（「〇〇の場合は△△にして」）
```

CLAUDE.md = 自分が書く「ルールブック」、Memory MD = Claude が自動で書く「業務日誌」。

### Git でセーブポイントを作る

大きな作業の前に GitHub Desktop 等で「作業前セーブ」としてコミットする。意図しない変更があってもセーブポイントまで戻れる。RPG のセーブ＆ロードと同じ発想。

<a name="token"></a>
## トークン効率化 — 使用制限に引っかからない技術

### なぜトークンが急速に消費されるのか

```
消費量の計算式: S × N(N+1) / 2
（S = 1回あたりの平均トークン数、N = メッセージ数）

1回あたり約500トークンの場合:
 5件:  約 7,500 トークン
10件:  約 27,500 トークン
20件:  約 105,000 トークン
30件:  約 232,000 トークン ← 1件目の31倍

実測: 総トークンの 98.5% が「過去の読み返し」に消費
```

### 即効習慣①  フォローアップより「編集して再生成」

```
❌ 悪い習慣（履歴が積み上がる）:
「違います、〇〇という意味です」→ 送信

✅ 良い習慣（履歴が置き換えられる）:
元のメッセージを「Edit」で修正 → 再生成
```

### 即効習慣② 15〜20 件で新しいセッションを開始

```
セッション終了の手順:
1. 「今の進捗・決定事項・次のタスクをファイルに書き出して」と指示
2. /clear でリセット
3. 書き出したファイルを冒頭に読み込ませて再開
```

1 時間以上放置した長期セッションも、再開せず新規開始を推奨（Anthropic 開発者の公式推奨）。

### 即効習慣③ 質問はまとめて1メッセージで送る

```
❌ 分けて送る（コンテキスト3回分消費）:
「この記事を要約して」→「主要ポイントを列挙して」→「見出しを提案して」

✅ まとめて送る（コンテキスト1回分）:
「この記事を要約し、主要ポイントを列挙して、見出しを3つ提案して」
```

まとめると Claude が全体像を把握でき、回答の質も上がることが多い。

### モデルを使い分ける

| モデル | 向いているタスク | コスト |
|--------|----------------|--------|
| **Haiku** | 文法チェック・箇条書き・短い翻訳・フォーマット | 低（50〜70% 節約） |
| **Sonnet** | 日常の作業・文章作成・中程度の分析 | 中 |
| **Opus** | 複雑な判断・深い推論が必要な場面 | 高（Sonnet の約2倍速で消費） |

> **Pro プランなら Sonnet 4.6 をデフォルトに設定**するのが Anthropic 開発者の公式推奨。Opus は同じ作業でもトークンが約2倍速で消費される。

### 思考モードはデフォルト OFF

Extended Thinking は深い推論が必要な場面でのみ有効にする。最初の回答が不十分だった場合だけ ON に切り替える。

```bash
# 長時間セッションのコスト抑制（Anthropic 公式推奨値）
export CLAUDE_CODE_AUTO_COMPACT_WINDOW=200000
```

### 使用タイミングを分散させる

Claude の制限は**ローリング5時間ウィンドウ**で管理される。

```
❌ 朝にまとめて全部使い切る → 午後が使えない
✅ 朝・昼・夜の3セッションに分散 → 常に枠が回復している状態
```

**ピーク時間帯（2026年3月26日以降）は同じ作業でも消費レートが上がる:**

```
太平洋時間 午前5時〜午前11時（平日）
東部時間   午前8時〜午後2時（平日）
日本時間   午後9時〜翌午前3時 がピーク帯に相当
```

重い処理は夕方・週末に実行すると制限に余裕が生まれる。

### 外部 AI に処理を振り分ける（ゼロトークンリサーチ）

「トークンをどう節約するか」ではなく「**どの処理をどの AI に振るか**」を設計する考え方。

```
① Claude Code が NotebookLM にノートブックを作成
② PDF・URL・YouTube 字幕などをアップロード
③ 分析は Google の Gemini が無料で処理
④ Claude Code は結果の整理と最終仕上げだけ担当
```

このワークフローをスキルとして登録しておけば、次回から自動で NotebookLM を活用するエージェントが起動する。

**注意**: `notebooklm-py` は非公式実装のため突然使えなくなるリスクがある。本番運用への依存は避け、「複数 AI で弱点を補い合う」設計思想として参考にする。

### プロジェクト機能でファイルをキャッシュする

同じ PDF を複数チャットにアップロードするとその都度トークンが消費される。プロジェクト機能を使えば一度アップロードするだけでキャッシュされ、繰り返し参照してもトークンを消費しない。

### Memory と User Preferences でセットアップコストをゼロにする

毎回新しいチャットで「私はマーケターで、カジュアルな文体で、短い段落が好きです」と説明し直すと、それだけで 3〜5 メッセージ分のトークンが消える。

```
Settings → Memory and User Settings に保存すべき内容:
- 自分の役割・職種
- コミュニケーションスタイル（カジュアル・フォーマルなど）
- よく使う出力フォーマットの好み
- 「Act as...」で毎回書いていた内容
```

一度保存すれば全チャットに自動適用される。

### Extra Usage（従量課金）を安全網として設定する

```
Settings → Usage → Extra Usage（Overage）をオン
月間上限金額を設定
→ 制限到達時に API レートの従量課金に移行
```

作業の途中で突然止まることを防ぐ安全網として機能する。

### 異常な消費を感じたら /feedback を実行する

小さなセッションでも異常なトークン消費を感じたら `/feedback` を実行することを Anthropic が推奨している。問題の報告が改善につながる。

<a name="maintenance"></a>
## 自動メンテナンス — Claude 自身に設定を点検させる

育てた設定は放置すると劣化する。CLAUDE.md の肥大化・allow ルールの蓄積・スキルの陳腐化・file-history の肥大化が進む。Claude Code 自身に定期点検させることで自動化できる。

### /maintenance スキルの作成

`.claude/skills/maintenance.md` を作成する:

````markdown
# /maintenance スキル — 定期メンテナンス

## 目的
Claude Code の設定・権限・スキル・ファイルを点検し、問題を報告・修正する。

## 実行手順

### フェーズ1: 権限の棚卸し
/permissions を実行し、以下を確認してレポートする:
- 不要と思われる「Always allow」ルール
- deny リストに含まれるべきなのに含まれていないコマンド
- 重複しているルール

### フェーズ2: CLAUDE.md の健全性チェック
各 CLAUDE.md ファイルを読み込み、以下を確認する:
- 削除しても動作が変わらない冗長な行
- 矛盾している指示
- 300行を超えているファイル（肥大化の警告）

### フェーズ3: セキュリティチェック
~/.claude/file-history/ を確認し、.env・.pem・.key など
機密ファイルのバックアップが含まれていないか確認する。
含まれている場合は削除前に確認を求める。

### フェーズ4: スキルの棚卸し
.claude/skills/ 内の全スキルを一覧表示し、
古いスキルや未使用スキルをリストアップする。

### フェーズ5: レポート出力
~/work/maintenance-report-[YYYYMMDD].md に保存する:

---
# メンテナンスレポート [日付]
## 🔴 要対応（緊急）
## 🟡 推奨対応
## 🟢 問題なし
## 実施した自動修正
---

## 注意事項
- ファイルの削除は必ず確認を求めてから実行する
- CLAUDE.md の修正前に内容を表示し、承認を得ること
- deny ルールの変更は慎重に行い、セキュリティが下がる変更は行わない
````

### /wrap-up スキル — セッション間の永続記憶を育てる

`.claude/skills/wrap-up.md` を作成する:

````markdown
# /wrap-up スキル — セッション終了・引き継ぎ

## 目的
セッション終了前に今回の学びと進捗を自動抽出・保存し、次回セッションに引き継ぐ。

## 実行手順
1. 今回うまくいったパターンを3〜5個抽出する
2. 今回 Claude が間違えたこと・修正が必要だったことを抽出する
3. 未完了のタスクと次にすべきアクションをリストアップする
4. 現在のプロジェクト状態をまとめる
5. 以下のファイルに保存する:
   - 引き継ぎ: ~/work/session-handoff-[YYYYMMDD-HHMM].md
   - 永続メモリ: ~/.claude/memory/learnings.md に追記

## 出力形式
---
# セッション引き継ぎ [日時]
## 今回完了したこと
## 次回やること（優先順）
## うまくいったパターン
## 注意点（次回避けること）
## 現在のプロジェクト状態
---

## 使い方
セッション終了前に必ず実行。
次回開始時: 「前回の引き継ぎファイルを読み込んで作業を再開して」
````

週単位で回すと、Claude Code が自分の好みやプロジェクト構造を覚えた状態でスタートできるようになる。

### 自動リマインダー（hooks を使う場合）

30日以上メンテナンスしていない場合に起動時にリマインダーを表示する:

```bash
#!/bin/bash
# ~/.claude/hooks/maintenance-reminder.sh

LAST_MAINTENANCE_FILE="$HOME/work/maintenance/.last-run"
DAYS_THRESHOLD=30

if [ -f "$LAST_MAINTENANCE_FILE" ]; then
  LAST_RUN=$(cat "$LAST_MAINTENANCE_FILE")
  DAYS_SINCE=$(( ($(date +%s) - LAST_RUN) / 86400 ))
  if [ "$DAYS_SINCE" -gt "$DAYS_THRESHOLD" ]; then
    echo "⚠️ メンテナンスから${DAYS_SINCE}日経過しています。/maintenance の実行を推奨します。"
  fi
fi
exit 0
```

```json
{
  "hooks": {
    "UserPromptSubmit": [
      { "type": "command", "command": "~/.claude/hooks/maintenance-reminder.sh" }
    ]
  }
}
```

| タイミング | アクション | 目的 |

| 毎回・セッション終了時 | `/wrap-up` | 学びを保存・次回引き継ぎ |
| 週1回 | 「file-history を削除して」 | 機密バックアップの削除 |
